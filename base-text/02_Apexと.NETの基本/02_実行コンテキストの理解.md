# 実行コンテキストの理解

## 学習の目的

この単元を完了すると、次のことができるようになります。

- Apex の呼び出しに使用する方法を理解する
- Salesforce オブジェクトのトリガーを記述する
- 開発者コンソールでコードを実行して実行コンテキストの動作を観察する
- ガバナ制限が設計パターンに与える影響を理解する
- 一括操作を使用することの重要性を理解する

> [!ポイント] この単元のゴール
>
> 「**実行コンテキストとは、コードが起動してから終了するまでの一区切り**」であり、その間に同じ**ガバナ制限**が適用される、という考え方が最重要です。あわせて「トリガーは一度に最大 200 件を受け取る」「ループ内で SOQL / DML を書かない（＝一括処理対応）」を押さえれば試験対策になります。

## 実行コンテキストとは?

ASP.NET ではコードはアプリケーションドメインのコンテキストで実行されますが、Lightning プラットフォームでは**実行コンテキスト**内で実行されます。これは**コードが実行されてから終了するまでの時間**を表し、その間に自分のコード以外のコードが実行されることもあります。

> [!用語] 実行コンテキスト（Execution Context）
>
> 1 つの起点（トリガー、匿名 Apex、Web サービスなど）から始まる一連の処理のまとまり。**この 1 つのまとまりに対して 1 セットのガバナ制限**が適用されます。トリガーが起動すると、そのトリガー・呼ばれたクラス・連動するフローやワークフローまで、すべてが同じ実行コンテキストとして数えられます。

### Apex を呼び出す方法

| 方法 | 説明 |
| --- | --- |
| **データベーストリガー** | カスタムまたは標準オブジェクトでの特定のイベントに対して呼び出される |
| **匿名 Apex** | 開発者コンソールやその他のツールでその場で実行されるコードスニペット |
| **非同期 Apex** | `future` / Queueable Apex、一括処理ジョブ、スケジュール実行 |
| **Web サービス** | SOAP または REST のインバウンド/アウトバウンド Web サービスを介して公開されるコード |
| **メールサービス** | 受信メールまたは送信メールを処理するために設定されるコード |
| **Visualforce または Lightning ページ** | Visualforce コントローラーや Lightning コンポーネントが、自動的に、またはユーザー操作（ボタンクリックなど）で Apex を実行する。Lightning コンポーネントはプロセスやフローからも実行できる |

Apex の呼び出し以外にも、ToDo 作成・メール送信・項目自動更新・アウトバウンドメッセージ送信などのアクションを宣言型機能でトリガーでき、これらも実行コンテキスト内で実行されます。

> [!注意] 「自分のコードだけ」が動くとは限らない
>
> 1 つの実行コンテキストには、自分のトリガーだけでなく**他の人が作ったトリガー・フロー・ワークフロー・入力規則**なども巻き込まれて動くことがあります。これらが同じガバナ制限を共有するため、「自分のコードは軽いのに制限に達した」という事態が起こり得ます。

デフォルトでは Apex は**システムコンテキスト**で実行され、すべてのオブジェクト・項目にアクセスできます（オブジェクト権限・FLS・共有ルールは適用されない）。`with sharing` を付けると現在のユーザーの共有ルールを考慮します。

> [!ポイント] システムコンテキストと with sharing
>
> - デフォルト（無指定 / `without sharing`）：**システムコンテキスト**で動き、共有ルールを無視してすべてのレコードにアクセスできる。
> - `with sharing`：実行ユーザーの**共有ルールを尊重**し、見えないレコードは扱えない。
>
> セキュリティ上、ユーザー操作起点のクラスでは `with sharing` を付けるのが基本です。

## トリガーの基本

SQL Server のトリガーと同様、Apex データベーストリガーは Salesforce のレコードに対するイベントの**前または後**にロジックを実行します。

> [!用語] トリガー（Trigger）
>
> レコードの保存（挿入・更新・削除など）に連動して**自動的に実行される Apex コード**。「取引先が登録されたら商談を自動作成する」のように、データ変更イベントをきっかけに処理を走らせます。

トリガー定義では次のイベントを 1 つ以上指定できます：`before insert` / `before update` / `before delete` / `after insert` / `after update` / `after delete` / `after undelete`。

基本構文は次のとおりです。

```apex
trigger TriggerName on ObjectName (trigger_events) {
   // code_block
}
```

### トリガーの種類

| 種類 | 使いどころ | レコードの編集 |
| --- | --- | --- |
| **before トリガー** | レコードがデータベースに保存される**前**に値を更新または検証する | レコードを**直接変更できる**（DML 不要） |
| **after トリガー** | システムが設定した項目値（`Id`、`LastModifiedDate` など）にアクセスする、他のレコードに影響を与える | トリガーのレコードは**参照のみ**（変更不可） |

> [!ポイント] before と after の使い分け（試験頻出）
>
> - 自分自身のレコードの値を**書き換えたい / 検証したい** → **before**（DML を呼ばずにそのまま変更できる）
> - 保存後に確定する **`Id` を使いたい**、または**別のレコードを作りたい** → **after**
>
> 「ShippingState を BillingState に合わせる」は before、「取引先作成後に商談を作る」は after が定石です。

> [!注意] before トリガーでの DML
>
> before トリガーのレコードはトリガー終了後にシステムが保存します。**DML を明示的にコールせずに** `Trigger.new` を変更でき、これらのレコードに対して DML を実行するとエラーになります。

> [!用語] DML（Data Manipulation Language）
>
> レコードを操作する命令。Apex では `insert` / `update` / `delete` / `upsert` / `undelete` がこれにあたります。データベースに変更を書き込む操作で、ガバナ制限のカウント対象です。

トリガーは、Salesforce のポイント & クリック自動化ツールで実現できないことが確実な場合の**最後の手段**として使います。コードなしでビジネスロジックを管理する強力なツール **Flow Builder** があり、かつてトリガーでしか実行できなかったタスクの多くにより適した自動化ツールが用意されています。

> [!ポイント] まず宣言型、最後の手段がトリガー
>
> Salesforce は「**Flow などの宣言型自動化で実現できないか先に検討し、無理な場合にだけトリガーを書く**」ことを推奨。試験でも「コードと自動化ツールのどちらを選ぶべきか」が問われます。安易なトリガー追加は不要な技術的負債をもたらします。

## 実行コンテキストのマーク

**新規取引先が入力されたら商談を作成する**トリガーを作ります。このトリガーはハンドラークラスのメソッドを呼ぶため、まずハンドラークラスを作成します。

> [!用語] トリガーハンドラーパターン（Trigger Handler Pattern）
>
> トリガー本体には「いつ・どのメソッドを呼ぶか」だけを書き、実際のロジックは別の**ハンドラークラス**に切り出す設計。トリガーを薄く保つことでテストや再利用がしやすくなり、ベストプラクティスとされています。

> [!手順] AccountHandler クラスを作成する
>
> 1. **[Setup（設定）]** から **[Developer Console（開発者コンソール）]** を選択する。
> 2. **[File（ファイル）] > [New（新規）] > [Apex Class（Apex クラス）]** を選択する。
> 3. クラス名として `AccountHandler` と入力し、**[OK]** をクリックする。
> 4. 既存のコードを削除して、次のスニペットを挿入する。
> 5. **Ctrl + S** キーを押してクラスを保存する。

```apex
public with sharing class AccountHandler {
    public static void CreateNewOpportunity(List<Account> accts) {
        for(Account acct : accts) {
            Opportunity opp = new Opportunity();
            opp.Name = acct.Name + ' Opportunity';
            opp.AccountId = acct.Id;
            opp.StageName = 'Prospecting';
            opp.CloseDate = System.Today().addMonths(1);
            insert opp;   // ← 注意：ループ内で DML を実行している（後で修正）
        }
    }
}
```

> [!注意] このコードには問題がある（あとで直す）
>
> 上のコードは `for` ループの**中**で `insert opp;` を実行しています。これは一括処理に対応しておらず、後述のガバナ制限に引っかかる可能性があります。後半で正しい書き方に修正します。

> [!ポイント] 1 オブジェクトにトリガーは 1 つ
>
> 同じオブジェクトに複数トリガーを作ると実行順序が保証されず管理が困難になります。**オブジェクトごとにトリガーは 1 つ**にし、ロジックはハンドラークラスへ切り出すのがベストプラクティスです。

次に取引先トリガーを作成します。

> [!手順] AccountTrigger トリガーを作成する
>
> 1. **[File（ファイル）] > [New（新規）] > [Apex Trigger（Apex トリガー）]** を選択する。
> 2. 名前に `AccountTrigger` と入力し、sObject として **[Account（取引先）]** を選択する。
> 3. **[Submit（送信）]** をクリックする。
> 4. 既存のコードを削除して、次のスニペットを挿入する。
> 5. **Ctrl + S** キーを押してトリガーを保存する。

```apex
trigger AccountTrigger on Account(before insert, before update, before
    delete, after insert, after update, after delete, after undelete) {
    if(Trigger.isAfter && Trigger.isInsert) {
        AccountHandler.CreateNewOpportunity(Trigger.new);
    }
}
```

> [!用語] トリガーコンテキスト変数（Trigger.xxx）
>
> トリガー内でだけ使える特別な変数。状況に応じた分岐に使います。
>
> | 変数 | 意味 |
> | --- | --- |
> | `Trigger.isBefore` / `Trigger.isAfter` | before か after か |
> | `Trigger.isInsert` / `isUpdate` / `isDelete` など | どのイベントか |
> | `Trigger.new` | 新しい値のレコードリスト（insert/update 時） |
> | `Trigger.old` | 変更前の値のレコードリスト（update/delete 時） |
>
> 上のトリガーは「**after かつ insert のときだけ**」ハンドラーを呼んでいます。

最後に、新規取引先を入力するユーザーをシミュレートする匿名コードを実行します。

> [!手順] 匿名 Apex で取引先を 1 件挿入する
>
> 1. **[Debug（デバッグ）] > [Open Execute Anonymous Window（実行匿名ウィンドウを開く）]** を選択する。
> 2. 既存のコードを削除し、次のスニペットを挿入する。
> 3. **[Open Log（ログを開く）]** が選択されていることを確認し、**[Execute（実行）]** をクリックする。
> 4. 新しいタブに実行ログが表示される。調査のため開いたままにする。

```apex
Account acct = new Account(
    Name='Test Account 2',
    Phone='(415)555-8989',
    NumberOfEmployees=50,
    BillingCity='San Francisco');
insert acct;
```

## 実行ログの調査

実行ログの最初の行には `EXECUTION_STARTED`、最後の行には `EXECUTION_FINISHED` がマークされます。**その間にあるすべてが実行コンテキストです。**

```text
EXECUTION_STARTED  ◀── 実行コンテキストの開始
 ├ CODE_UNIT_STARTED  （匿名 Apex の起動）
 │   └ insert acct;  ← 取引先を挿入
 ├ CODE_UNIT_STARTED  （BeforeInsert イベントのコード）
 ├ CODE_UNIT_STARTED  （AfterInsert イベント＝トリガーが商談を作成）
 │   └ ワークフロールールがあればここでも実行される
 ▼
EXECUTION_FINISHED  ◀── 実行コンテキストの終了
        ↑
   この間すべてが「同じ 1 つの実行コンテキスト」＝同じガバナ制限を共有
```

ログには、匿名 Apex 起動・`BeforeInsert`・`AfterInsert`（トリガーが商談作成）の各 `CODE_UNIT_STARTED` が並びます。ワークフロールールを作成していればそれらも表示されます。**これらはすべて同じ実行コンテキスト下で動作するため、同じガバナ制限セットが適用されます。**

ガバナ制限が重要なのは、Salesforce が**マルチテナント環境**だからです。1 つの組織がリソースを消費しすぎてシステム全体がクラッシュしないように、こうした制限が不可欠です。

> [!用語] マルチテナント（Multi-Tenant）
>
> 1 つの物理基盤を多数の顧客（テナント）が共有するアーキテクチャ。マンションのように建物（インフラ）は共有しつつ、各部屋（組織）のデータは隔離されます。1 人が資源を使いすぎると全員に影響するため、利用量に上限（ガバナ制限）が設けられます。

> [!用語] ガバナ制限（Governor Limits）
>
> マルチテナント環境で 1 つの組織が資源を独占しないよう設けられた**実行ごとの上限値**。1 実行コンテキストあたりの SOQL クエリ数、DML 件数、CPU 時間などが制限されます。超えると例外（`System.LimitException`）が発生し、処理全体がロールバックされます。

## 制限への対応

最もよく考慮することになるのが、**SOQL クエリ数と DML ステートメント数**の制限です。

> [!注意] 制限値は変わる
>
> 制限は数多くあり、メジャーリリースごとに変化する傾向があります（緩くなることもよくある）。必ず「リソース」の「実行ガバナと制限」リンクで**最新の制限**を確認してください。

## 一括処理

コードを 1 件のレコード処理用に設計するのは Lightning プラットフォームでは大きな間違いです。

> [!用語] 一括処理対応（Bulkification／バルク化）
>
> 1 件ずつではなく、**複数レコードをまとめて処理できるように**コードを書くこと。具体的には「SOQL や DML をループの外に出し、レコードはリストに溜めてから一括で処理する」ことを指します。

**Apex トリガーが一度に受信できるレコードは最大 200 個**です。同期での SOQL クエリ上限は **100**、DML ステートメント上限は **150** です。ループ内で SOQL / DML を実行するトリガーが一括処理で起動すると制限エラーになります。

> [!ポイント] 覚えておきたい数値（試験頻出）
>
> | 項目 | 同期処理の上限 |
> | --- | --- |
> | トリガーが一度に受け取るレコード数 | **最大 200 件**（1 バッチ） |
> | SOQL クエリの発行数 | **100** |
> | DML ステートメントの発行数 | **150** |
>
> 200 件にループ内で `insert` すると DML が 200 回発行され、上限 150 を超えて**制限エラー**になります。

制限に達するまでは正常に動くため、後から「一括処理対応」へ設計し直すことになりがちで、これは最初の設計より時間がかかります。最初から一括処理に対応するよう設計しましょう。

> [!例] なぜループ内 DML が危険か
>
> 取引先を 200 件まとめて登録すると、トリガーは 200 件を 1 バッチで受け取ります。ループ内で 1 件ずつ `insert opp;` すると **DML が 200 回**呼ばれ、上限 150 を超えてエラー。代わりに 200 件分の商談をリストに溜めて**最後に 1 回だけ `insert opps;`** すれば、DML は 1 回で済みます。

前に作成したハンドラーは一括パターンを使っていないため、修正します。ループ内でリスト変数に書き出し、**リストの内容をまとめて挿入**します。

> [!手順] AccountHandler を一括処理対応に修正する
>
> 1. **[File（ファイル）] > [Open（開く）]** を選択する。
> 2. エンティティ種別として **[Classes（クラス）]**、エンティティとして **[AccountHandler]** を選択する。
> 3. **[Open（開く）]** をクリックする。
> 4. 既存のコードを削除して、次のスニペットを挿入する。
> 5. **Ctrl + S** キーを押してクラスを保存する。

```apex
public with sharing class AccountHandler {
    public static void CreateNewOpportunity(List<Account> accts) {
        List<Opportunity> opps = new List<Opportunity>();   // 商談を溜めるリスト
        for(Account acct : accts) {
            Opportunity opp = new Opportunity();
            opp.Name = acct.Name + ' Opportunity';
            opp.AccountId = acct.Id;
            opp.StageName = 'Prospecting';
            opp.CloseDate = System.Today().addMonths(1);
            opps.add(opp);     // ← ループ内ではリストに追加するだけ
        }
        if(opps.size() > 0) {
            insert opps;       // ← ループの外で 1 回だけまとめて挿入
        }
    }
}
```

> [!ポイント] 一括処理対応の鉄則
>
> - **SOQL も DML もループの外**に出す。
> - ループ内ではコレクション（List / Map）に**溜めるだけ**。
> - 挿入前に `if(list.size() > 0)` で空チェックすると無駄な DML を避けられる。

修正できたので、200 レコードの負荷を処理できることを**単体テスト**で確認します。

> [!用語] 単体テスト / `Test.startTest()`・`Test.stopTest()`
>
> `@isTest` を付けたクラス/メソッドがテストコードです。`Test.startTest()` と `Test.stopTest()` で囲んだ区間は、**ガバナ制限がリセットされた新しいコンテキスト**として実行されるため、データ作成とテスト対象の処理を切り分けて検証できます。

> [!手順] 一括テストクラスを作成・実行する
>
> 1. **[File（ファイル）] > [New（新規）] > [Apex Class（Apex クラス）]** を選択する。
> 2. クラス名として `AccountTrigger_Test` と入力し、**[OK]** をクリックする。
> 3. 既存のコードを削除して、次のスニペットを挿入する。
> 4. **Ctrl + S** キーを押してクラスを保存する。
> 5. **[Test（テスト）] > [New Run（新規実行）]** を選択する。
> 6. TestClass として `AccountTrigger_Test`、テストメソッドとして `TestCreateNewAccountInBulk` を選択する。
> 7. **[実行]** をクリックする。
> 8. **[Test（テスト）]** タブで、テストがエラーなしで完了したこと（緑のチェックマーク）を確認する。

```apex
@isTest
private class AccountTrigger_Test {
    @isTest static void TestCreateNewAccountInBulk() {
        // テスト用データの準備：取引先を 200 件作成
        List<Account> accts = new List<Account>();
        for(Integer i=0; i < 200; i++) {
            Account acct = new Account(Name='Test Account ' + i);
            accts.add(acct);
        }
        // テストの実行
        Test.startTest();
            insert accts;
        Test.stopTest();
        // 取引先が 200 件挿入されたことを検証
        List<Account> verifyAccts = [SELECT Id FROM Account];
        Assert.areEqual(200, verifyAccts.size());
        // 商談も 200 件挿入されたことを検証
        List<Opportunity> verifyOpps = [SELECT Id FROM Opportunity];
        Assert.areEqual(200, verifyOpps.size());
    }
}
```

> [!注意] 単体テストはまだ深く扱わない
>
> 単体テストはまだ詳しく説明していませんが、いくつかのキーワード（`@isTest`、`Assert.areEqual` など）があるだけで、Lightning プラットフォームでも .NET と同様に動作します。詳細は「リソース」のリンクを参照してください。

## もうひとこと...

Apex でも `try-catch-finally` で例外を処理しますが、コードの実行場所に応じて `catch` や `rollback` の扱いが変わることがあります（ベストプラクティスは「リソース」参照）。

Lightning プラットフォームには**アプリケーション変数やセッション変数のようなものはありません**。クラス間でデータを保持するには**静的変数**を使いますが、その動作は .NET とは異なります。

> [!ポイント] 静的変数のスコープは「1 実行コンテキスト」
>
> .NET のアプリケーション/セッション変数は複数リクエストをまたいで生き続けますが、Lightning プラットフォームの静的変数が情報を保持できるのは**1 つの実行コンテキスト内に限定**されます。実行コンテキストが終われば値は消えます（再帰トリガーの制御などに使われる）。トリガー呼び出し間でデータを保持したい場合は別のオプションが必要です。

特に管理パッケージ開発者は、制限対応で多くのトレードオフを考慮する必要があります。本格的な Apex 開発には「リソース」の高度な Apex のリンクを参照してください。

> [!ポイント] この単元の頻出ポイントまとめ
>
> - **実行コンテキスト = `EXECUTION_STARTED` から `EXECUTION_FINISHED` まで**。この単位でガバナ制限が適用される。
> - トリガー・フロー・ワークフローなど**連動するすべてが同じ制限を共有**する。
> - トリガーは一度に**最大 200 件**を受け取る。SOQL は **100**、DML は **150**（同期）。
> - **SOQL / DML をループ内に書かない**（一括処理対応）。
> - **before** はレコードの加工・検証（DML 不要）、**after** は `Id` 利用・関連レコード作成。
> - **1 オブジェクト 1 トリガー**＋ハンドラークラスがベストプラクティス。
> - 静的変数は**1 実行コンテキスト内**でしか値を保持しない。

## ハンズオン Challenge（+500 ポイント）

> [!まとめ] あなたの Challenge：項目を変更してから挿入するトリガーを作成する
>
> **Account（取引先）の項目を変更してからレコードを挿入する Apex トリガー**を作成します。レコードの挿入前に起動され、`ShippingState` 項目と `BillingState` 項目の値が同じであることを確認します。
>
> **1. ハンドラークラス**
> - `public static` メソッドを含む Apex クラスを作成する
> - 名前：`AccountTriggerHandler`
> - メソッド名：`CreateAccounts`
> - メソッドは Account（取引先）オブジェクトの**リストを受け入れる**必要がある
> - 保存する前に、各 Account レコードの `ShippingState` 項目の値が `BillingState` 項目と同じになるように更新する
>
> **2. Apex トリガー**
> - 名前：`AccountTrigger`
> - sObject：`Account`（取引先）
> - 起動：**レコードの挿入前（before insert）**
> - 新しいレコードのコレクションで `AccountTriggerHandler.CreateAccounts()` を呼び出す
> - `isBefore` と `isInsert` トリガーコンテキスト変数を使用する
>
> **3. テストクラス**
> - 名前：`AccountTriggerTest`
> - `BillingState` が `CA` の Account レコードを **200 件挿入**する
> - 挿入後、200 件すべてのレコードの `ShippingState` が `CA` になっていることを確認する
>
> この Challenge の完了を確認する前に、開発者コンソールの **[Run All（すべて実行）]** 機能を使用して、少なくとも 1 回テストクラスを実行する。

> [!ポイント] Challenge 攻略のヒント
>
> - 自レコードの値を保存前に書き換えるので、トリガーは **before insert** が正解（after では `Trigger.new` を変更できない）。
> - トリガー内で `if(Trigger.isBefore && Trigger.isInsert)` と分岐し、`Trigger.new` をハンドラーに渡す。
> - ハンドラーは渡された List をループし、各レコードの `ShippingState = acct.BillingState;` を代入するだけ（before なので DML は不要）。
