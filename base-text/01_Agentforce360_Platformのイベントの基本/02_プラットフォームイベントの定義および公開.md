# プラットフォームイベントの定義および公開

## 学習の目的

この単元を完了すると、次のことができるようになります。

- プラットフォームイベントを定義する。
- Apex でプラットフォームイベントメッセージを公開する方法を説明する。
- Apex メソッドを使用してイベントを公開する。
- プロセスまたはフローのクリック操作でイベントを公開する。
- REST API を使用して sObject を挿入することでイベントを公開する。

> [!ポイント] この単元のゴール
>
> プラットフォームイベントは「**カスタムオブジェクトのように定義し、いろいろな方法で公開できる**」のが特徴です。定義の手順（`__e` サフィックス・公開動作）、4 つの公開方法（Apex / フロー / REST API / Pub/Sub API）、`EventBus.publish()` の使い方を押さえれば Challenge も突破できます。

## プラットフォームイベントの定義

前の単元の **Cloud News 通信社**のニュースイベントを保持するプラットフォームイベント定義を作成しましょう。

> [!手順] Cloud News プラットフォームイベントを定義する
>
> 1. **[設定（Setup）]** の **[クイック検索（Quick Find）]** に `Platform Events` と入力し、**[プラットフォームイベント（Platform Events）]** を選択する。
> 2. **[新規プラットフォームイベント]** をクリックする。
> 3. **[表示ラベル（Label）]** に `Cloud News` と入力する。
> 4. **[表示ラベル（複数形）（Plural Label）]** に `Cloud News` と入力する。
> 5. **[説明（Description）]** に `Cloud news events deliver news at your fingertips` と入力する。
> 6. **[公開動作（Publish Behavior）]** はデフォルトの **[すぐに公開（Publish Immediately）]** のままにする。
> 7. **[保存（Save）]** をクリックする。
> 8. **[カスタム項目 & リレーション]** 関連リストで **[新規（New）]** をクリックする。
> 9. **[テキスト]** を選択し **[次へ（Next）]** をクリックする。
> 10. **[項目の表示ラベル／名前（Field Label/Name）]** に `Location`（場所）と入力する。
> 11. **[長さ（Length）]** に `100` と入力する。その他はデフォルト、[説明] は空白のまま **[保存（Save）]** をクリックする。
> 12. ステップ 8 〜 11 を繰り返して、次の 2 項目を追加する。

| 項目の表示ラベル／名前 | データ型 |
| --- | --- |
| 緊急 | チェックボックス |
| News Content | テキストエリア（ロング） |

サポートされるデータ型は次のとおりです。

| サポートされるデータ型 | 用途の例 |
| --- | --- |
| チェックボックス | 緊急フラグ（true/false） |
| Date（日付） | 発生日 |
| 日付/時間 | 発生日時 |
| Number（番号） | 数量・金額 |
| テキスト | 場所名（長さ指定あり） |
| テキストエリア（ロング） | 本文・説明 |

> [!ポイント] サポートされるデータ型は限定されている
>
> 使えるのは上記の **6 種類のみ**。カスタムオブジェクトにある「選択リスト」「数式」「参照関係」などは使えません。イベントは「データを運ぶシンプルなメッセージ」なので型もシンプルに絞られています。

## イベント保持と ReplayId システム項目

**大規模プラットフォームイベントは 72 時間**保存されます（Spring ’19 以前定義の標準規模イベントは 24 時間）。保存されたイベントは Pub/Sub API でイベントバスから取得でき、すべて取得するか、**再実行 ID（ReplayId）**を基準に指定して取得できます。イベントメッセージは一時保持されますが、**SOQL / SOSL でクエリできず**、レポート・リストビュー・検索の UI でも使用できません。

> [!注意] 新しいイベントはすべて「大規模」
>
> 新しく定義されたイベントはデフォルトで**大規模イベント**になります（標準規模は現在定義不可）。保持期間は基本 **72 時間**と覚えれば十分です。

> [!用語] SOQL / SOSL / ReplayId
>
> **SOQL（Salesforce Object Query Language）**＝レコードを検索・取得するクエリ言語（SQL に類似）。**SOSL（Salesforce Object Search Language）**＝複数オブジェクト横断のテキスト検索言語。イベントメッセージはこれらでクエリできません。
> **ReplayId（再実行 ID）**＝各イベントに割り当てられる不透明な ID で、ストリーム内の位置を表す。登録者はこの値を保存し、**接続障害からの復旧時**に「ここから再取得」と指定して欠落イベントを取得できます（連番の保証はなく、自分で計算して他イベントを参照することもできません）。

## API 参照名サフィックス

イベント作成時、システムが **`__e` サフィックス**を付加して API 参照名を作ります（例：`Cloud_News__e`）。Apex・REST API・Pub/Sub API など、**プログラムで参照する場合は常に API 参照名を使用**します。

> [!ポイント] サフィックスの使い分け
>
> | サフィックス | 意味 | 例 |
> | --- | --- | --- |
> | `__c` | カスタムオブジェクト／カスタム項目 | `Location__c` |
> | `__e` | プラットフォーム**イベント**（**e**vent） | `Cloud_News__e` |
>
> 「イベントなら `__e`」と即答できるようにしましょう。

## プラットフォームイベントの公開動作とトランザクション

**公開動作**は、イベントが公開されるタイミング（公開コール直後か、トランザクション完了後か）を指定します。

> [!用語] トランザクション（Transaction）
>
> 一連のデータベース操作を「ひとまとまり」として扱う単位。すべて成功すれば確定（**コミット**）、途中で失敗すればすべて取り消し（**ロールバック**）。「コミット後に公開」は、このコミット成功時だけイベントを出す動作です。

- **すぐに公開（Publish Immediately）**：デフォルトで推奨。**トランザクションの成否に関係なく公開コール実行時点で公開**するため、データのコミット前に登録者が受信する場合がある。公開者と登録者が独立する場合や、記録（ログ）目的に適する。
- **コミット後に公開（Publish After Commit）**：登録者が受信時に**コミット済みデータを必要とする場合のみ**選択。トランザクションが正常コミットした場合のみ公開し、**失敗時は公開されない**。例：イベントでタスクレコードを作成し、登録した別プロセスが**そのレコードを探す**場合。

| 比較項目 | すぐに公開（Publish Immediately） | コミット後に公開（Publish After Commit） |
| --- | --- | --- |
| 公開のタイミング | 公開コールの実行直後 | トランザクションのコミット成功後 |
| トランザクション失敗時 | それでも公開される | 公開されない |
| 登録者がコミット済みデータに依存 | しない場合に適する | する場合に選ぶ |
| デフォルト／推奨 | **デフォルト・推奨** | 必要な場合のみ |
| DML 制限の扱い | 個別の公開制限（150 件）に含まれる | Apex DML ステートメント制限（1 件）に含まれる |

> [!ポイント] 2 つの公開動作の選び分け（頻出）
>
> - **登録者が「公開元のデータ」を見る必要がない** → [すぐに公開]（デフォルト）
> - **登録者が「公開元がコミットしたレコード」を参照する必要がある** → [コミット後に公開]
>
> 「失敗しても公開したい＝すぐに公開／失敗したら公開したくない＝コミット後に公開」と覚えましょう。

## イベントの公開

Salesforce Platform 上では **Apex メソッド**または **Flow Builder** で公開でき、外部アプリケーションは **Salesforce API（Pub/Sub API、REST などのデータ API）**で公開できます。

```text
  ┌─────────────────── イベント公開の方法 ───────────────────┐
  │                                                          │
  │  Salesforce Platform 上                外部アプリケーション  │
  │  ┌────────────────────┐              ┌────────────────┐ │
  │  │ Apex                │              │ Pub/Sub API     │ │
  │  │  EventBus.publish() │              │                 │ │
  │  ├────────────────────┤              ├────────────────┤ │
  │  │ フロー／プロセス       │              │ REST/SOAP/Bulk  │ │
  │  │ （クリック操作）       │              │ （sObject 挿入） │ │
  │  └────────────────────┘              └────────────────┘ │
  │                          ▼ 公開 ▼                         │
  │                    ┌──────────────┐                       │
  │                    │ イベントバス   │                       │
  │                    └──────────────┘                       │
  └──────────────────────────────────────────────────────────┘
```

## Apex を使用したイベントメッセージの公開

イベントメッセージを公開するには、**イベントのインスタンスを作成して `EventBus.publish()` に渡します**。

> [!用語] EventBus.publish() / DML
>
> **`EventBus.publish()`**：プラットフォームイベントを公開する Apex メソッド。渡したインスタンスをキューに入れ**非同期で**公開する。結果は `Database.SaveResult` で返り、`isSuccess()` で成否を確認。**公開失敗しても例外は発生しない**（戻り値で確認）。
> **DML（Data Manipulation Language）**：レコードの挿入・更新・削除などデータ操作命令の総称（`insert`、`update`、`delete` 等）。イベント公開も DML に近い扱いで、戻り値 `Database.SaveResult` は DML 結果と同じ型です。

次の例は `Cloud_News__e` のイベントを 1 つ作成して公開し、成否をチェックします。`isSuccess()` が `true` なら公開要求がキューに入り**非同期で公開**され、`false` ならエラーが `Database.Error` で返ります（例外は発生しません）。

> [!手順] 開発者コンソールでスニペットを実行する
>
> 1. クイックアクセスメニューから **[開発者コンソール（Developer Console）]** をクリックする。
> 2. **[Debug]** | **[Open Execute Anonymous Window]** をクリックする。
> 3. 内容を下記スニペットに置き換えて **[Execute（実行）]** をクリックする。

```apex
// イベントのインスタンスを作成し、newsEvent 変数に格納する
Cloud_News__e newsEvent = new Cloud_News__e(
           Location__c='Mountain City',
           Urgent__c=true,
           News_Content__c='Lake Road is closed due to mudslides.');
// イベントを公開するメソッドを呼び出す
Database.SaveResult sr = EventBus.publish(newsEvent);
// 公開結果を確認する
if (sr.isSuccess()) {
    System.debug('Successfully published event.');
} else {
    // 失敗した場合はエラー内容を 1 件ずつ出力する
    for(Database.Error err : sr.getErrors()) {
        System.debug('Error returned: ' +
                     err.getStatusCode() +
                     ' - ' +
                     err.getMessage());
    }
}
```

**複数のイベントをまとめて公開**するには、イベントのリストを `EventBus.publish()` に渡します。出力は `Database.SaveResult` の**配列**（公開イベントごとに 1 つ）。エラーで公開できないものを除いて**部分的に公開**でき、例外は発生しません（部分完了オプション指定時の `Database.insert()` に似た動作）。

```apex
// 公開するイベントオブジェクトを保持するリスト
List<Cloud_News__e> newsEventList = new List<Cloud_News__e>();
// イベントオブジェクトを作成する
Cloud_News__e newsEvent1 = new Cloud_News__e(
           Location__c='Mountain City',
           Urgent__c=true,
           News_Content__c='Lake Road is closed due to mudslides.');
Cloud_News__e newsEvent2 = new Cloud_News__e(
           Location__c='Mountain City',
           Urgent__c=false,
           News_Content__c='Small incident on Goat Lane causing traffic.');
// イベントオブジェクトをリストに追加する
newsEventList.add(newsEvent1);
newsEventList.add(newsEvent2);
// イベントを公開するメソッドを呼び出す
List<Database.SaveResult> results = EventBus.publish(newsEventList);
// イベントごとに公開結果を確認する
for (Database.SaveResult sr : results) {
    if (sr.isSuccess()) {
        System.debug('Successfully published event.');
    } else {
        for(Database.Error err : sr.getErrors()) {
            System.debug('Error returned: ' +
                        err.getStatusCode() +
                        ' - ' +
                        err.getMessage());
        }
    }
}
```

> [!例] 単一公開と一括公開の使い分け
>
> - **1 件だけ公開** → `EventBus.publish(newsEvent)`。戻り値は `Database.SaveResult` 1 つ。
> - **複数まとめて公開** → リストにためて `EventBus.publish(newsEventList)`。戻り値は `List<Database.SaveResult>`。大量公開はループ内で 1 件ずつ呼ばず**リストにまとめて 1 回で公開**するのがベストプラクティス（ガバナ制限対策）。

キューに登録されたイベントはシステムリソースが使えるようになると**非同期に公開**され、保持期間中イベントバスに保持されます。まれに内部システムエラーで非同期公開が失敗することがあり、最終的な公開結果は **Apex 公開コールバック**で確認できます（詳細は『プラットフォームイベント開発者ガイド』参照）。

> [!注意] イベントの割り当て（ガバナ制限）
>
> 組織には**定義できるイベント数**と **1 時間に公開できるイベント数**の割り当てがあります。
>
> - **[コミット後に公開]** のイベントは、各実行が **Apex DML ステートメント制限**に対し 1 DML（`Limits.getDMLStatements()` で確認）。
> - **[すぐに公開]** のイベントは、各実行が **`EventBus.publish()` コール 150 件**の個別公開制限に含まれる（`Limits.getPublishImmediateDML()` で確認）。
>
> ガバナ制限とは、マルチテナント環境で 1 つの処理がリソースを独占しないよう Apex 実行に設けられた上限です（SOQL 100 件、DML 150 件など）。イベント公開にも適用されます。

## クリック操作を使用したイベントメッセージの公開

コーディングなしで公開するには、**Flow Builder の [レコード作成] 要素**をプラットフォームイベントのインスタンス（`Cloud_News__e`）作成用に設定し、値を入力します。`Urgent__c` など Boolean（チェックボックス）項目には `{!$GlobalConstant.True}` を使用します。

> [!例] ノーコードで公開できる場面
>
> 管理者が「特定レコードが更新されたらイベントを飛ばしたい」場合、Apex を書かずにフローの [レコード作成] でイベント（`__e` オブジェクト）を作るだけで公開できます。開発者でなくても公開できるのが宣言型ツールの強みです。

## Salesforce API を使用したイベントメッセージの公開

外部アプリケーションは **Pub/Sub API** または **SOAP / REST / Bulk API などのデータ API** でイベントを公開します。データ API では **sObject を挿入するのと同じ方法**で公開し、Apex 同様に非同期で公開されます。成功状況 `true` が返ると、公開要求がキューに配置されます。

例として、`Cloud_News__e` レコードを挿入してイベントを公開します。次は REST API で `Cloud_News__e` のイベントを 1 つ作成する例です。

> [!手順] Postman で REST API からイベントを公開する
>
> はじめての場合は先に「Quick Start: Connect Postman to Salesforce」プロジェクトで Salesforce APIs コレクションを設定・フォークしておきます。
>
> **DE 組織を Postman に接続する**
>
> 1. Trailhead DE 組織にログインする。
> 2. Postman でワークスペースを選び、Salesforce API コレクションのフォークに移動する。
> 3. **[Authorization]** タブ最下部の **[Get New Access Token]** をクリックする。
> 4. **[Allow]** をクリックする。
> 5. **[Manage Access Tokens]** ダイアログで **[instance URL]** をクリップボードにコピーする。
> 6. **[Use Token]** をクリックする。
> 7. **[Variables]** タブの `_endpoint` 行 **[CURRENT VALUE]** 列に、コピーしたインスタンス URL を貼り付け **[Save]** をクリックする（ドキュメントペインを閉じないと [Save] が出ない場合がある）。
>
> **接続をテストする**
>
> 1. Salesforce APIs コレクションのフォークを選択する。
> 2. **[REST]** を展開する。
> 3. **[GET Limits]** を選択して **[Send]** をクリックする。
> 4. [Status] に **[Status: 200 OK]** と出れば成功。出ない場合はトークンを取得し直す。
>
> **REST API コールでイベント通知を公開する**
>
> 1. フォークで **[REST]** をクリックする。
> 2. **[sObject]** を展開する。
> 3. **[POST SObject Create]** をクリックする。
> 4. **[Params]** タブの **[Path Variables]** の `SOBJECT_API_NAME` に `Cloud_News__e` と入力する。
> 5. **[Body]** タブに次の JSON ボディを貼り付ける。
> 6. **[Send]** をクリックする。

POST 要求のリクエストボディ（JSON）の例:

```json
{
   "Location__c" : "Mountain City",
   "Urgent__c" : true,
   "News_Content__c" : "Lake Road is closed due to mudslides."
}
```

イベント投稿後に Salesforce が返す応答の例:

```json
{
  "id" : "e00xx0000000001AAA",
  "success" : true,
  "errors" : [
       {
            "statusCode": "OPERATION ENQUEUED",
            "message": "08ffc869-b9f8-4cff-a4ba-8dff9b3dd6cb",
            "fields": []
       }
   ]
}
```

> [!ポイント] 応答の `OPERATION ENQUEUED` の意味
>
> `success: true` で返る `statusCode: "OPERATION ENQUEUED"` は「**公開要求がキューに入った（受け付けられた）**」の意味で、「配信完了した」わけではありません。イベント公開は非同期なので、API がすぐ返す応答は「受付完了」を表します。

## 試験対策：押さえておきたい追加ポイント

> [!ポイント] 定義・公開のよくある出題
>
> - イベントの API 参照名サフィックスは **`__e`**（例：`Cloud_News__e`）。
> - 公開は **`EventBus.publish()`**。戻り値は **`Database.SaveResult`**（複数公開なら配列）。`isSuccess()` で成否確認、**失敗しても例外は出ない**。
> - 公開は常に**非同期**。応答 `OPERATION ENQUEUED` は「キュー登録済み」。
> - 公開動作は **[すぐに公開]（デフォルト・推奨）** と **[コミット後に公開]**（コミット成功時のみ公開）。
> - 公開方法は **Apex / フロー・プロセス / REST API（sObject 挿入）/ Pub/Sub API**。外部は API、Salesforce 内は Apex かフロー。
> - サポートされる項目データ型は **チェックボックス・日付・日付/時間・番号・テキスト・テキストエリア（ロング）** の 6 種類。

## ハンズオン Challenge（+500 ポイント）

> [!まとめ] あなたの Challenge：プラットフォームイベントの定義
>
> 2 つの項目を持つプラットフォームイベントを作成します。注文アプリケーションがこのイベントを送信し、注文発送を Salesforce アプリケーションに通知します。**この Challenge ではプラットフォームイベントのみを作成します。**
>
> **Order Event（注文イベント）プラットフォームイベントを作成する**
>
> | 設定項目 | 値 |
> | --- | --- |
> | 表示ラベル | `Order Event`（注文イベント） |
> | 表示ラベル（複数形） | `Order Events`（注文イベント） |
> | オブジェクト名 | `Order_Event` |
> | 説明 | `Information about a customer order`（顧客注文に関する情報） |
> | 公開動作 | Publish Immediately（すぐに公開） |
>
> **[Order Number（注文番号）] 項目を作成する**
>
> | 設定項目 | 値 |
> | --- | --- |
> | データ型（Field Type） | Text（テキスト） |
> | 表示ラベル | `Order Number`（注文番号） |
> | 文字数 | `10` |
> | 名前 | `Order_Number` |
>
> **[Has Shipped（発送済み）] 項目を作成する**
>
> | 設定項目 | 値 |
> | --- | --- |
> | データ型 | Checkbox（チェックボックス） |
