# 適切な検索ソリューションの選択

## 学習の目的

この単元を完了すると、次のことができるようになります。

- SOSL と SOQL の相違点を説明する。
- 検索に使用できる API プロトコルを特定する。

> [!ポイント] この単元のゴール
>
> 「**データの場所がわかっているなら SOQL、わからないなら SOSL**」が最大の勘所です。あわせて検索インデックスのしくみ、SOQL/SOSL を送信する API プロトコル（REST・SOAP・Apex）、その他の検索 API（推奨レコード・統合検索）の役割を押さえれば試験対策は十分です。

---

## Salesforce 流の検索

**検索**は最もよく使われる Salesforce 機能で、何千件ものデータから求める 1 件を見つける手段です。このモジュールでは検索のしくみ、カスタム検索ソリューションの要否判断、API での実装、結果の最適化を学びます。

> [!用語] 検索インデックス（Search Index）
>
> レコード本文をそのまま保存せず、検索しやすいよう加工・整理した「索引」のこと。「この単語はどのレコードに出るか」を高速に引ける。組織の通常データベースとは**別に**作られる。

レコードを作成・更新すると、検索エンジンがデータのコピーを作り、コンテンツを**トークン**に分割。トークンは元レコードへのリンクと共に検索インデックスに保存されます。

> [!用語] トークン（Token）・トークン化（Tokenization）
>
> 文章を検索しやすい**最小単位（おもに単語）に分割したもの**がトークン、その処理がトークン化。「bunny slippers」は「bunny」「slippers」の 2 トークンに分かれ、語順違いや部分一致でも探せる。

ユーザー側から見た検索の流れは次のとおりです。

```text
ユーザーが検索語を入力 (1)
        │
        ▼
検索エンジンが検索語をトークンに分割 (2)
        │
        ▼
トークンを検索インデックスのレコード情報と照合 (3)
        │
        ▼
関連するレコードを関連性でランク付け (4)
        │
        ▼
ユーザーがアクセスできる結果を返す (5)
```

## 検索インデックスのしくみ

トークンを使う理由は、**どの結果を返すかをスマートに決定できる**ため。検索エンジンはスペル修正・ニックネーム・見出語化・シノニムグループなどの高度な機能を適用し、検索語のバリエーションを含むレコードも拾えます。

> [!用語] 見出語化（Lemmatization）
>
> 検索語の**活用形・変化形をまとめて同じ語として扱う**しくみ。「add」で検索すると「adding」「added」も一致する。

> [!用語] シノニムグループ（Synonym Group・同義語グループ）
>
> 意味が同じ語をまとめ「どれで検索しても他もヒットする」設定。「USA」「United States」「米国」を同グループにすると、どの語でも同じレコードが見つかる。

もう一つの利点が**関連性ランキング**。検索語の頻度・順序・ユニーク性、レコードアクティビティ、アクセス権限などを使って結果に順位を付けます。

| 比較項目 | データベース検索（SOQL） | 検索インデックス検索（SOSL） |
| --- | --- | --- |
| 一致のしかた | 完全一致が基本 | あいまい一致（バリエーションも拾う） |
| スペル修正 | 効かない | 効く |
| 語順違い | 一致しない | 順不同一致で一致する |
| 活用形・同義語 | 効かない | 見出語化・シノニムで効く |
| 結果の並び | 自分で指定 | 関連性で自動ランク付け |

> [!例] 「bunny slippers（ウサギのスリッパ）」で検索すると
>
> - **データベース検索**：「bunny slippers」と完全一致するレコードのみ。
> - **検索インデックス検索**：「rabbit slippers」「bunny slipper（単数形）」などの類似語句、「slippers bunny」の語順違い、スペル間違いまで拾い、**関連性順**に返す。

## カスタム検索ソリューションが必要になる場面

カスタム検索ソリューションが必要なのは、組織で**標準 UI の代わりにカスタム UI を使う場合**です（顧客向け知識ベース、従業員向け社内商品サイトなど）。カスタム UI を構築する場合は、カスタム検索で Salesforce 検索の高度な機能の一部を利用できます。

> [!ポイント] カスタム検索が「適している」ケースを覚える
>
> 試験頻出。答えは **標準 UI ではなくカスタム UI（自家製の知識ベースや顧客向けサイトなど）を構築する場合**。「標準 Lightning Experience を使い始めた」「検索バーにブランド設定を置きたい」だけでは理由にならない。

## API を使用した検索への接続

レコードを見つける主な API として、まず 2 つを覚えます（他の 2 つは後述）。

- **Salesforce Object Query Language（SOQL）**
- **Salesforce Object Search Language（SOSL）**

> [!用語] SOQL（ソークル）/ SOSL（ソッスル）
>
> - **SOQL**：Salesforce 版の SQL。`SELECT` 文と同等で**組織のデータベースを直接検索**。「どのオブジェクトのどの項目か」が分かるとき。
> - **SOSL**：プログラムから**検索インデックスにテキスト検索**を行う言語。スペル修正・見出語化・関連性ランキングが使える。「どこにあるか分からないが、この言葉を含むものを探す」とき。

SOQL と SOSL は**構文も異なります**。

### SOQL と SOSL の使い分け

| 観点 | SOQL を使う | SOSL を使う |
| --- | --- | --- |
| データの場所 | **どのオブジェクト・項目にあるか分かっている** | **どこにあるか分からない** |
| 主な用途 | 関連データの取得・集計・並べ替え | テキストの全文検索 |
| 検索インデックス | 使わない（DB を直接検索） | 使う（あいまい一致が効く） |
| 数値・日付・チェックボックス | 取得しやすい | テキスト中心 |

データの場所が**わかっていて**、次をしたいなら **SOQL**：

- 1 つ、または関連付けられた複数オブジェクトからデータを取得する。
- 条件を満たすレコード数を数える。
- 結果を並び替える。
- 数値・日付・チェックボックス項目からデータを取得する。

データの場所が**不明で**、次をしたいなら **SOSL**：

- 項目内にあると分かっている特定語のデータを取得する（複数語をトークン化して高速に多くの結果を返せる）。
- 関連・無関連を問わず複数のオブジェクト・項目を効率的に取得する。
- ディビジョン機能で特定ディビジョンのデータを効率的に取得する。

> [!例] SOQL か SOSL かを判断する
>
> - 「**取引先**の年間売上 1 億円以上を売上順で取りたい」→ オブジェクトも項目も明確なので **SOQL**。
> - 「『田中』が取引先・取引先責任者・リードのどこかに含まれるレコードを横断して探す」→ 場所不明＋複数オブジェクト横断なので **SOSL**。

> [!ポイント] 最頻出の使い分け
>
> 「SOSL は SOQL とどう違うか」は頻出。覚えるべきは次の 3 点（および「上記のすべて」）。
> - **構文が異なる**。
> - **SOSL は組織データベースではなく検索インデックスを検索する**。
> - **SOSL は、データがどのオブジェクトにあるか分からない場合により効率的**。

## さらに 2 つの API 種別

### 推奨レコード API（Suggested Records）

自動推奨・インスタント結果・先行入力（typeahead）とも呼ばれます。

> [!用語] 先行入力（Typeahead）
>
> 検索ボックスに数文字入力した時点で候補を即表示する機能。Google の入力候補と同じ。最後まで打たなくても目的のレコードに到達できる。

> [!例] 推奨レコードの体験
>
> ランニングシューズ販売サイトで「trail running」と入力し始めると、タイトルに検索語を含む記事候補がその場で表示される。

**Search Suggested Records** および **Search Suggested Articles** の REST メソッドで実現します（詳細は後の単元）。

### Salesforce 統合検索（Federated Search）

ユーザーがアクセスするレコードが **Salesforce 外部に保存されている**場合のソリューションです。

> [!用語] Salesforce 統合検索（Federated Search）
>
> グローバル検索ボックスから**外部の検索エンジン**を検索できるしくみ。Classic・コンソール・Lightning Experience を開いたまま、外部ソースの結果を Salesforce の検索結果に直接表示できる。**Salesforce 統合検索コネクタ**で実現し、コネクタは **OpenSearch 仕様**で構築されるため、同標準準拠の任意の検索エンジンに接続できる。

> [!注意] 統合検索では Salesforce の高度な機能は効かない
>
> 統合検索は **Salesforce 検索インデックスを使わない**ため、スペル修正・見出語化・関連性ランキングは適用されない。結果は**外部検索プロバイダーのしくみに従って**返る。

## プロトコルを使用したクエリの送信

検索クエリは **API プロトコルで送信**しないと実行されません。主なプロトコルは **REST・SOAP・Apex** の 3 つです。

> [!用語] API プロトコル（REST / SOAP / Apex）
>
> クエリを Salesforce に送って実行させる通信手段。**REST**・**SOAP** は外部から HTTP で呼ぶ Web サービス、**Apex** は Salesforce 内部で動く言語。SOQL/SOSL という「クエリ言語」を、これら「運び役」に乗せて実行する。

SOQL クエリと SOSL 検索のプロトコル対応は次のとおりです。

| 目的 | REST | SOAP | 内容 |
| --- | --- | --- | --- |
| SOQL クエリの実行 | **Query** | **query()** | 指定オブジェクトに対し SOQL を実行し、条件一致データを返す |
| SOSL 検索の実行 | **Search** | **search()** | 組織データに対し SOSL テキスト文字列検索を行う |

SOSL/SOQL を使いたくない場合は、REST の **Parameterized Search** を検討します。

> [!用語] Parameterized Search（パラメーター化検索）
>
> URL 内に SOSL のクエリ本文を書く代わりに、**URL のパラメーターで検索条件を指定**する REST の検索方法。SOSL の構文を書かずに検索したいときに便利。

Apex では**角括弧 `[ ]` で囲んで** SOQL/SOSL をその場で使えます。`Search` クラスで動的 SOSL を実行したり、`Search` 名前空間で検索・推奨結果を取得したりできます。

```apex
// Apex 内に SOQL を直接埋め込む例（インライン SOQL）。角括弧 [ ] で囲む。
List<Account> accts = [SELECT Id, Name FROM Account WHERE Name = 'Acme'];

// Apex 内に SOSL を直接埋め込む例（インライン SOSL）。
// FIND で検索語、IN で検索対象、RETURNING で返すオブジェクトと項目を指定。
List<List<SObject>> results = [FIND 'Acme' IN ALL FIELDS RETURNING Account(Id, Name)];
```

> [!ポイント] プロトコルと API の対応を覚える
>
> - **SOQL → Query（REST）/ query()（SOAP）**
> - **SOSL → Search（REST）/ search()（SOAP）**
> - **Apex** では `[ ]`（角括弧）で SOQL/SOSL をインライン実行。
> - REST で SOSL/SOQL を書きたくないときは **Parameterized Search**。

## 試験対策：押さえておきたい追加ポイント

> [!ポイント] この単元の頻出ポイントまとめ
>
> - **SOQL = データベースを直接検索**。場所が分かり、取得・集計・並べ替えをするとき。
> - **SOSL = 検索インデックスを検索**。場所不明／複数オブジェクト横断で全文検索したいとき。あいまい一致が効く。
> - SOSL と SOQL は **構文も異なる**。
> - カスタム検索が適するのは **カスタム UI を構築する場合**（外部知識ベース等）。
> - プロトコルは **REST・SOAP・Apex**。SOQL→Query/query()、SOSL→Search/search()。
> - **統合検索は検索インデックスを使わない**ので高度な検索機能は効かない。

> [!注意] SOQL と SOSL を取り違えない
>
> 「複数オブジェクトをまたいで取得」という言葉だけで SOSL と即断しない。**関連付けられた複数オブジェクト**を関係をたどって取得するのは SOQL でも可能。決め手は「**場所が分かるか（SOQL）／全文テキスト検索したいか（SOSL）**」。
