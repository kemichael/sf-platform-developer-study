# 開発者コンソールの使用開始

宇宙船が Salesforce で駆動するように、開発者コンソールを使えば会社の Salesforce 組織を操縦できます。まずは「開発者コンソールとは何か」から押さえましょう。

> [!ポイント] この単元のゴール
>
> **開発者コンソールとは何か / いつ使うか / どうアクセスするか / ワークスペースで作業をどう整理するか**。この4点を押さえれば、以降の単元（コード編集・ログ分析・チェックポイント・SOQL/SOSL）に進めます。

---

## 学習の目的

この単元を完了すると、次のことができるようになります。

- 開発者コンソールとは何かを説明する。
- 開発者コンソールをいつ使うべきかを判断する。
- 開発者コンソールにアクセスする。
- ワークスペースを作成・切り替えして作業を整理する。

---

## 開発者コンソールとは?

開発者コンソールは、組織のアプリケーションを作成・デバッグ・テストできる**統合開発環境（IDE）**です。

> [!用語] 開発者コンソール（Developer Console）
>
> Salesforce 組織のブラウザー上で動く開発ツール。コードの作成・編集、デバッグ、テスト、クエリ実行をまとめて行える「開発作業のコックピット」。インストール不要で、組織にログインすればすぐ使えます。

> [!用語] IDE（統合開発環境）
>
> 「エディター＋デバッガー＋実行環境」が一体になり、開発作業を1つの画面で行えるツールの総称。

開発者コンソールは次のタスクに対応するワンストップソリューションです。

- Apex クラス・トリガー・Aura コンポーネント・Visualforce ページなどを作成・編集する。
- 組織で作成したパッケージを参照する。
- デバッグログを生成し、さまざまな視点から分析する。
- Apex コードをテストしてエラーがないことを確認する。
- Apex コードにチェックポイントを設定し、エラーを特定・解決する。
- SOQL / SOSL クエリを記述・実行してレコードを検索・作成・更新する。

> [!用語] Apex（エイペックス）
>
> Salesforce 独自のオブジェクト指向プログラミング言語。Java に似た文法で、組織にビジネスロジックを追加するときに使います。本モジュールでは深く立ち入りません。

> [!用語] SOQL / SOSL
>
> **SOQL**（Salesforce Object Query Language）は**特定オブジェクトのレコードを読み込む**クエリ言語。**SOSL**（Salesforce Object Search Language）は**複数オブジェクトをまたいでテキスト検索**する言語（単元5で詳しく学びます）。

> [!注意] 開発者コンソールでは LWC は作れない
>
> Spring '19（API 45.0）以降、Lightning コンポーネントには **Lightning Web コンポーネント（LWC）** と **Aura コンポーネント** の2モデルがあり、同じページ上で共存できます。本コンテンツで扱うのは **Aura コンポーネント** です。**開発者コンソールでは LWC は開発できません**（試験頻出）。

## どんな場合に開発者コンソールを使用するか

Salesforce 組織の制御には、**VS Code 向け Salesforce 拡張機能**、**開発者コンソール**、**サードパーティツール**などの選択肢があります。

開発者コンソールは **1 組織に接続**・**ブラウザーベース**で動作し、変更がすぐ有効になります。インストール不要ですぐ試したいときに最適です。複数組織への接続、ファイルの比較・同期、バージョン管理が必要なら VS Code 向け Salesforce 拡張機能が向いています。

### 開発ツールの比較

| 比較項目 | 開発者コンソール | VS Code 向け Salesforce 拡張機能 |
| --- | --- | --- |
| 接続できる組織 | **1 つだけ** | **複数** |
| 動作場所 | **ブラウザー上** | ローカル PC（要インストール） |
| インストール | **不要** | 必要 |
| 変更の反映 | **すぐ有効** | デプロイ操作が必要 |
| バージョン管理 | **なし** | あり（Git など） |
| ファイル比較・同期 | なし | あり |
| 向いている場面 | すぐに試したい・軽い修正 | 本格開発・チーム開発 |

> [!ポイント] コンソール vs VS Code 拡張機能（頻出）
>
> 「**1 組織・ブラウザーベース・インストール不要・バージョン管理なし** = 開発者コンソール」「**複数組織・比較／同期／バージョン管理あり** = VS Code 拡張機能」という対比は定番。開発者コンソールには**バージョン管理や競合解決の機能がない**ため、共有組織では**他ユーザーのコードを上書きしないよう注意**します。

## 開発者コンソールへのアクセス

> [!手順] 開発者コンソールを開く
>
> 1. 組織にログインする。
> 2. **クイックアクセスメニュー**、または画面右上の**あなたの名前**をクリックする。
> 3. メニューから **[開発者コンソール（Developer Console）]** をクリックする。

この操作は **Lightning Experience** でも **Salesforce Classic** でも利用できます（メニューの場所が少し異なります）。

初めて開くと、画面はおおよそ次の構成です。

```text
┌─────────────────────────────────────────────┐
│  メニューバー（File / Edit / Debug / Test …）  │
├─────────────────────────────────────────────┤
│                                             │
│   (1) メインペイン                            │
│       ＝ ソースコードエディター                 │
│       コードを記述・表示・変更する              │
│                                             │
├─────────────────────────────────────────────┤
│   (2) タブペイン                              │
│       ＝ ログ / エラー / クエリ などを表示       │
│       組織のレコード操作クエリもここで記述       │
└─────────────────────────────────────────────┘
```

- **メインペイン (1)**：ソースコードエディター。コードを記述・表示・変更する。
- **タブペイン (2)**：ログ・エラー等を表示し、レコード操作クエリも記述する。

## ワークスペースの設定

開発者コンソールを開いたら、次は**ワークスペース**の設定です。

### ワークスペースとは?

宇宙船で月を目指しているのに火星着陸の情報が出たら困ります。ワークスペースは、各開発タスクに必要な内容だけを表示できるよう情報を整理する仕組みです。

> [!用語] ワークスペース（Workspace）
>
> メインパネルの各タブにまとめた**一連のリソース**（コード・テスト・ログなど）。「作業セットをまるごと保存しておく仕組み」とイメージしてください。プロジェクト単位で作っておけば、切り替えるときに必要なものだけを一括表示できます。

> [!例] ワークスペースの使い分け
>
> エンジニアリングチーム向けのコードを書きながらナビゲーションチーム向けの情報も見たいときは、ワークスペースを2つ作り、それぞれに関連リソースだけを含めます。

初回使用時は**デフォルトのワークスペース**が表示されます。独自のものも簡単に作れます。

### 独自のワークスペースの設定

> [!手順] ワークスペースを新規作成・切り替えする
>
> 1. **[Workspace] | [New Workspace]** を選択する。
> 2. ワークスペースに名前を付ける。
> 3. 新しいワークスペースで、Apex クラス・Aura コンポーネント・Visualforce ページなどを作成する。
> 4. 切り替えるには **[Workspace] | [Switch Workspace]** を選択する。

```text
[Workspace]
   ├─ New Workspace      … 新しいワークスペースを作る
   └─ Switch Workspace   … 既存のワークスペースに切り替える
        ├─ プロジェクトA（コード / テスト / ログ）
        └─ プロジェクトB（コード / テスト / ログ）
```

## 試験対策：押さえておきたい追加ポイント

> [!ポイント] この単元の頻出ポイント整理
>
> - **開発者コンソール = IDE**。コードの作成・デバッグ・テスト・クエリ実行ができる。
> - 接続は **1 組織のみ**、**ブラウザーベース**、**インストール不要**、**バージョン管理なし**。
> - **LWC は開発者コンソールでは作れない**。Aura コンポーネントは作れる。
> - 複数組織・比較／同期／バージョン管理が必要なら **VS Code 向け Salesforce 拡張機能**。
> - **ワークスペース**は「タブにまとめたリソースの集合」。プロジェクト単位で整理・切り替えできる。

## テスト

この単元を完了するには、テストのすべての質問に正しく解答する必要があります。（+100 ポイント）

> [!まとめ] 確認テスト
>
> **問1：開発者コンソールを使用してできることはどれですか?**
> - A. Apex コードを記述して編集する。
> - B. Visualforce ページを作成して編集する。
> - C. Aura コンポーネントを作成して編集する。
> - D. デバッグログを確認する。
> - **E. 上記のすべて。** ← Apex・Visualforce・Aura の作成編集とログ確認をすべて行える。
>
> **問2：開発者コンソールを説明するものはどれですか?**
> - A. 複数の組織に接続され、ブラウザーベースで動作する。
> - **B. 1 つの組織に接続され、ブラウザーベースで動作する。** ← 1 組織・ブラウザーベースが正解。
> - C. 1 つの組織に接続され、バージョン管理を備えている。
> - D. 複数の組織に接続されるが、バージョン管理を備えていない。
>
> **問3：ワークスペースマネージャーを使用してできることはどれですか?**
> - A. ワークスペースを作成して、プロジェクトごとにリソースを整理する。
> - B. 組織にデータを格納するためのカスタムオブジェクトを作成する。
> - C. ワークスペースを切り替える。
> - D. 上記のすべて。
> - **E. A と C の両方** ← ワークスペースの作成・整理と切り替えができる（カスタムオブジェクト作成は対象外）。
