# DML を使用したレコードの変更

## 学習の目的

この単元を完了すると、次のことができるようになります。

- 開発者コンソールのクエリ結果グリッドを使ってレコードを挿入・更新・削除する。
- DML（データ操作言語）とは何か、SOQL との役割の違いを説明する。
- 主要な DML ステートメント（insert / update / upsert / delete / undelete / merge）を区別する。
- DML を実行する 2 つの方法（DML ステートメントと Database クラスメソッド）の違いを説明する。

> [!ポイント] この単元のゴール
>
> SOQL がデータの「**読み取り**」だったのに対し、**DML はデータの「書き込み」**（追加・更新・削除）を担います。試験対策の2点。
> - DML ステートメントは **insert / update / upsert / delete / undelete / merge** の6種類（`unmerge` は存在しない）。
> - 実行方法は2つ。**DML ステートメント**（`insert acct;`）と **Database クラスメソッド**（`Database.insert(acct, false)`）。後者だけが**部分的な成功**を許可できる。

---

## クエリ結果グリッドでデータを操作する

開発者コンソールの **[クエリ結果（Query Results）]** ペインは結果を表示するだけでなく、**クエリグリッド**としても機能します。グリッドをクリックし、下のボタンでデータを挿入・更新・削除できます。

> [!用語] クエリ結果グリッド（Query Results グリッド）
>
> 開発者コンソールで SOQL の実行結果を表形式（グリッド）で表示する領域。表示だけでなく行の追加・編集・削除ができる「ミニ表計算ソフト」のように使え、コードを書かずにレコードを操作できます。

> [!手順] グリッドからレコードを挿入する
>
> 1. **[設定（Setup）]** メニューから **[開発者コンソール（Developer Console）]** を選択する。
> 2. **[クエリエディター（Query Editor）]** をクリックする。
> 3. 既存のコードをすべて削除して、次のクエリを挿入する。
>    ```sql
>    SELECT Id, Name, Industry FROM Account
>    ```
> 4. **[実行]** をクリックする。組織内のすべての取引先と**合計行数**が表示される。この値に注目しておく。
> 5. **[Insert Row（行を挿入）]** をクリックする。新しい行がグリッドの最上部に追加される。
> 6. 新しい行をダブルクリックして名前と業種を入力する。ただし **ID 値は空白のまま**にする。
> 7. **[Save Rows（行を保存）]** をクリックする。ID 列に値が自動的に表示される。
> 8. **[Refresh Grid（グリッドを更新）]** をクリックする。合計行数が 1 つ増えていることを確認する。

ID 列に値が自動表示されるのは、**ID が主キーで値をシステムが生成する**ためです。保存直後の合計行数は元のままで、グリッドを更新するまで件数表示は変わりません。

> [!例] なぜ ID を空白にするのか
>
> 新規レコードの ID は **Salesforce が保存時に自動採番**します（SQL の自動インクリメント主キーや IDENTITY 列と同じ発想）。`insert` のときは ID を指定せず、保存後に18文字の ID が返ってくる、と覚えましょう。

## DML とは

Salesforce では SOQL を使ってデータを**照会**しますが、データを**管理**する場合には **DML** を使います。

> [!用語] DML（Data Manipulation Language：データ操作言語）
>
> レコードの**追加・更新・削除・復元**などを行う仕組み。SOQL が「読み取り専門（SELECT）」なのに対し、DML は「書き込み専門」です。SQL では `SELECT` も `INSERT/UPDATE/DELETE` も同じ SQL の中にありますが、Salesforce では**読み取り＝SOQL／書き込み＝DML**と役割が分かれています。

```text
   ┌─────────────────────────────────────────┐
   │            Salesforce のデータ操作          │
   ├──────────────────┬──────────────────────┤
   │   読み取り（照会）   │   書き込み（管理）        │
   │   SOQL / SOSL     │   DML                  │
   │   SELECT / FIND   │  insert/update/delete… │
   └──────────────────┴──────────────────────┘
```

Salesforce は緊密に統合されているため、**データソースの管理（接続文字列・トランザクション管理など）を気にせず**すぐにデータ操作ができます。これが DML の操作が同等の SQL よりずっと簡単な理由です。

## DML ステートメント

| ステートメント | 説明 |
| --- | --- |
| `insert` | 新規レコードを挿入する。 |
| `update` | 既存のレコードを更新する。 |
| `upsert` | **特別な操作**。1つのステートメント内で新規作成または既存更新を行う。指定された項目で既存判定し、未指定なら `Id` を使用。 |
| `delete` | 既存のレコードを削除する。 |
| `undelete` | 1つ以上の既存のレコードをごみ箱から復元する。 |
| `merge` | **特別な操作**。同じオブジェクト種別の最大3つのレコードを1つにマージし、他は削除、関連レコードの親を変更する。 |

> [!ポイント] DML ステートメントは6種類（頻出・暗記）
>
> **insert / update / upsert / delete / undelete / merge** の6つ。
> 試験では「有効な DML ステートメントでないものはどれか」が問われ、**`unmerge` は存在しない**ため引っかけ選択肢になります。

> [!用語] upsert（アップサート）
>
> 「update + insert」の合成語。レコードが**あれば更新、なければ新規作成**する操作。一致判定には `Id` または**外部 ID 項目**を使い、外部システム連携で重複を作らず同期するのに重宝します。

> [!用語] merge（マージ）
>
> 重複した最大3件のレコードを1件に統合する操作。残りはごみ箱へ移し、関連する子レコードは統合先の親に付け替えます。

> [!例] insert と upsert の違い
>
> - `insert`：常に**新規**として追加。何度実行すれば重複が増える。
> - `upsert`：キー（Id か外部 ID）で照合し、**あれば更新／なければ追加**。連携処理で「重複させたくない」ときに使う。

## DML を実行する 2 つの方法

### 1. DML ステートメント自体を使用する

SOSL の単元でデータ挿入に使った方法で、**最も簡単**です。

```apex
// 取引先を追加する
Account acct = new Account(
    Name='Test Account',
    Phone='(225)555-8989',
    NumberOfEmployees=10,
    BillingCity='Baton Rouge'
);
insert acct;
```

> [!注意] DML ステートメントは「全件成功か全件失敗」
>
> DML ステートメント（`insert acct;` の形）は、リストを一括処理する際に**1件でも失敗すると操作全体がロールバックされ、例外が発生**します。「一部だけ成功」は許されません。

### 2. Database クラスメソッドを使用する

`Database` クラスメソッドの利点は、省略可能な **`allOrNone` パラメーター**で**部分的な成功**を許可できることです。`false` に設定すると、一部レコードでエラーが発生しても**成功したレコードはコミット**され、失敗分はエラーが返ります（**例外を発生させません**）。

```apex
// 取引先を追加する
Account acct = new Account(
    Name='Test Account',
    Phone='(225)555-8989',
    NumberOfEmployees=10,
    BillingCity='Baton Rouge'
);
// 第2引数 false で「部分的な成功」を許可する
Database.SaveResult[] results = Database.insert(acct, false);
```

> [!用語] allOrNone パラメーター
>
> `Database.insert(records, allOrNone)` の第2引数。`true`（既定）なら「全件成功か全件失敗（例外）」、`false` なら「成功分はコミットし、失敗分はエラー情報を返す（例外を出さない）」挙動になります。

> [!用語] Database.SaveResult
>
> `Database` メソッドが返す結果オブジェクト。レコードごとに**成功・失敗・エラー内容**を保持し、`result.isSuccess()` / `result.getErrors()` で1件ずつ確認できます。

## 2つの方法の比較

> [!ポイント] DML ステートメント vs Database クラスメソッド（頻出）
>
> | 観点 | DML ステートメント（`insert acct;`） | Database メソッド（`Database.insert(acct, false)`） |
> | --- | --- | --- |
> | 部分的な成功 | **不可**（全件成功 or 全件失敗） | **可能**（`allOrNone=false` で指定） |
> | エラー時の挙動 | **例外を発生**させる | `false` なら例外を出さず**結果配列**を返す |
> | 戻り値 | なし（成功した sObject に Id が入る） | `Database.SaveResult[]` などの結果配列 |
> | 書きやすさ | 最もシンプル | やや記述量が多い |
>
> ポイント：**「部分的な成功を許可できる／エラーを発生させないでおける」のは Database メソッドだけ**。`allOrNone` の既定は `true`（DML ステートメントと同様に例外を発生）で、部分成功を使いたいときだけ明示的に `false` を渡します。

## 試験対策：押さえておきたい追加ポイント

> [!まとめ] DML の要点整理
>
> - **読み取り＝SOQL／書き込み＝DML** の役割分担。
> - DML ステートメントは **insert / update / upsert / delete / undelete / merge** の6種類。`unmerge` は存在しない。
> - `upsert`＝あれば更新・なければ追加（Id か外部 ID で照合）。`merge`＝最大3件を1件に統合。
> - 実行方法は2つ：**DML ステートメント**（簡単・全件成功/失敗・例外あり）と **Database メソッド**（`allOrNone` で部分成功を制御・結果配列を返す）。
> - **部分的な成功を許可できるのは Database メソッドだけ**。`allOrNone` の既定は `true`。
> - DML はガバナ制限の対象。ループ内 DML を避けてバルク化する。
