# 効率的なクエリの作成

## 学習の目的

この単元を完了すると、次のことができるようになります。

- Salesforce クエリオプティマイザーがクエリのパフォーマンスを最適化するしくみを理解する。
- セレクティブクエリの使用がクエリのパフォーマンスに及ぼす影響を理解する。
- クエリ実行プランツールを使用して検索クエリを評価する方法を理解する。

> [!ポイント] この単元のゴール
>
> 「**遅いクエリ＝ガバナ制限に引っかかりやすいクエリ**」です。鍵は **セレクティブ（選択的）なクエリ**。要点は3つ。
> - 速さの秘密は **インデックス（索引）**。インデックス付き項目を `WHERE` で使うとセレクティブになる。
> - **null 照会・否定演算子・先頭ワイルドカード・テキスト項目への比較演算子**はインデックスを無効化する（避ける）。
> - **クエリ実行プランツール**で `TableScan`（全件走査）か `Index`（索引利用）かを確認できる。

---

## 制限（ガバナ制限）

効率的なクエリはパフォーマンスを高め、**ガバナ制限**の問題も防ぎます。マルチテナント環境ではパフォーマンスの悪いクエリがシステムをクラッシュさせる最大要因のため、ガバナ制限が設けられています。

> [!用語] ガバナ制限（Governor Limits）
>
> 1人のユーザーがリソースを使いすぎないよう Salesforce が課す**実行時の上限**（例: 1トランザクションで SOQL は最大100クエリ、取得レコードは最大50,000件）。超えると例外が発生し処理が中断します。効率の悪いクエリは制限に引っかかりやすくなります。

---

## Salesforce クエリオプティマイザー

Salesforce はバックエンドに Oracle を使いますが、**独自のクエリオプティマイザー**でコストベースにクエリを評価します。評価には収集した**統計**を使い、多くは週次で収集、一部は1時間ごとにキャッシュされます。

> [!用語] クエリオプティマイザー（Query Optimizer）
>
> クエリを「どの経路（インデックス利用か全件スキャンか）で実行すれば最速か」を自動判断する仕組み。統計情報をもとに各経路の**コスト**を見積もり、最良の実行プランを選びます（交通警官のようにクエリを適切なインデックスへ転送）。

```text
   SOQL / SOSL クエリ
          │
          ▼
 ┌──────────────────────┐
 │  クエリオプティマイザー   │  ← 統計をもとに各パスのコストを算定（交通警官）
 └──────────┬───────────┘
       ┌────┴────┐
       ▼         ▼
  インデックス利用   全件スキャン（TableScan）
   （速い／低コスト）   （遅い／高コスト）
```

## ベストプラクティス

### セレクティブクエリの作成

`WHERE` 句に項目を含めて返すデータを減らすのは良いことですが、**すべての項目が同じではありません**。一部は `WHERE` で使うとクエリが非常に高速になる「パワー項目」で、その秘密は**インデックス**です。

> [!用語] セレクティブクエリ／インデックス
>
> **セレクティブクエリ**は `WHERE` 句で**インデックス付き項目**を使い、全件走査せずに必要なレコードだけを絞り込めるクエリ（使えないものは「非セレクティブ」）。**インデックス（索引）**は、本の巻末索引のように項目の値から目的のレコードへ素早くたどり着く内部データ構造で、検索を劇的に速くします。

すべての標準・カスタムテーブルで、特定の項目は**自動的にインデックスが付けられます**。

| インデックス付き項目 | 説明 |
| --- | --- |
| `Id` | 一意の18文字の項目。オブジェクトの主キー。 |
| `Name`（名前） | テキストベースの項目。 |
| `OwnerId` | オブジェクトの所有者への参照。 |
| `CreatedDate` | レコードが作成された日時。 |
| `SystemModStamp` | 最終更新日を含む参照のみの項目。インデックスが付くが、類似の `LastModifiedDate` には付かないため、クエリではこちらを使う。 |
| `RecordType`（レコードタイプ） | RecordType の ID。 |
| 主従項目 | 主従関係を示す外部キー項目。 |
| 参照項目 | 参照関係を示す外部キー項目。 |
| 一意の項目 | 作成時に一意とマークすると自動でインデックスが付く。 |
| 外部 ID 項目 | 外部 ID としてマークできるカスタム項目。主にインテグレーション用。 |

これらを `WHERE` 句で使うとクエリが**セレクティブとみなされ**、完全スキャンではなくインデックスが使われやすくなります（大規模データで極めて重要）。

> [!ポイント] 自動でインデックスが付く主な項目（頻出）
>
> `Id` / `Name` / `OwnerId` / `CreatedDate` / `SystemModStamp` / `RecordType` / 主従・参照（外部キー）項目 / 一意項目 / 外部 ID 項目。
> 試験では「自動的にインデックス付けされる項目はどれか」が問われ、上記はすべて該当します。なお `LastModifiedDate` にはインデックスが付かないため、最終更新日の絞り込みには**インデックスの効く `SystemModStamp`** を使います。

> [!注意] カスタムインデックスの依頼
>
> **Salesforce サポートにカスタムインデックスを依頼**できます。サポートケースを作成し、対象項目を使った SOQL クエリを含めます。

### インデックスを使ってもセレクティブでなくなる例外

インデックス付き項目を使っても、クエリ内の操作で非セレクティブになり完全スキャンが行われることがあります。次は常に避けます。

1. **null 行に対するクエリ**
   ```sql
   SELECT Id, Name FROM Account WHERE Custom_Field__c = null
   ```
2. **否定的な絞り込み演算子**（`!=`、`NOT LIKE`、`EXCLUDES` など）
   ```sql
   SELECT CaseNumber FROM Case WHERE Status != 'New'
   ```
3. **先頭のワイルドカード**
   ```sql
   SELECT Id, LastName, FirstName FROM Contact WHERE LastName LIKE '%smi%'
   ```
4. **比較演算子を使用したテキスト項目**（`>`、`<`、`>=`、`<=`）
   ```sql
   SELECT AccountId, Amount FROM Opportunity WHERE Order_Number__c > 10
   ```

> [!ポイント] セレクティブを壊す4つの落とし穴（頻出・暗記）
>
> | 避けるべき操作 | 例 | なぜダメか |
> | --- | --- | --- |
> | null の照会 | `WHERE Field__c = null` | null はインデックス対象外になりやすい |
> | 否定演算子 | `!=`, `NOT LIKE`, `EXCLUDES` | 「該当しないもの」は索引で絞れない |
> | 先頭ワイルドカード | `LIKE '%smi%'` | 先頭が不定だと索引をたどれない |
> | テキスト項目への比較演算子 | テキスト項目に `>`, `<` | テキストは大小比較の索引が効かない |

> [!例] 同じ結果でも書き方で速さが変わる
>
> `WHERE Status != 'New'`（否定演算子・非セレクティブ）と `WHERE IsClosed = true`（インデックス付きチェックボックス・セレクティブ）は、同じケースを返しても実行プランが全く異なります。「何を除外するか」ではなく「何に一致するか」で書くのがコツです。

## クエリ実行プランツール

開発者コンソールにはクエリオプティマイザーの動作を確認できるツールがあります。既定では無効なので有効化します。

> [!手順] クエリ実行プランツールを有効にする
>
> 1. **[設定（Setup）]** → **[開発者コンソール（Developer Console）]** を選択する。
> 2. **[ヘルプ（Help）]** → **[設定（Preferences）]** を選択する。
> 3. **[Enable Query Plan（クエリプランを有効化）]** を `true` に設定する。
> 4. **[保存]** をクリックする。
> 5. **[クエリエディター（Query Editor）]** タブで **[実行（Execute）]** の横に **[Query Plan（クエリプラン）]** ボタンが表示されることを確認する。

> [!手順] パフォーマンスの悪いクエリ（TableScan）を確認する
>
> 1. **[クエリエディター（Query Editor）]** タブをクリックする。
> 2. 既存クエリを削除し、次を挿入する。
>    ```sql
>    SELECT Id, CaseNumber FROM Case WHERE Status != 'New'
>    ```
> 3. **[Query Plan（クエリプラン）]** をクリックする。
> 4. **`TableScan`（全件走査）が実行される**ことが表示される（良くない）。

> [!手順] インデックスが効くクエリと比較する
>
> 1. **[クエリエディター（Query Editor）]** タブをクリックする。
> 2. 既存クエリを削除し、次を挿入する。
>    ```sql
>    SELECT Id, CaseNumber FROM Case WHERE IsClosed = true
>    ```
> 3. **[Query Plan（クエリプラン）]** をクリックする。
> 4. 今回は**インデックスを使用できることを示す行**が表示される。

> [!用語] TableScan（テーブルスキャン／全件走査）
>
> インデックスを使わず、テーブルのレコードを**先頭から1件ずつすべて**調べる方式。数百万件規模では非常に遅く、ガバナ制限の原因になります。クエリプランで `TableScan` と出たら改善の余地ありのサインです。

どちらのクエリも取得行数は同じですが、**選択した項目と演算子で実行プランが大きく異なります**。開発組織では差が小さく見えても、数百万件規模では重要になります。

## もうひとこと...

**非決定性数式項目**を使ったクエリは常に避けます。

> [!用語] 非決定性数式項目（Nondeterministic Formula Field）
>
> `TODAY()` や他オブジェクト参照、積み上げ集計など、**時間や関連レコードの変化で値が変わる**数式項目。インデックスを付けられず、`WHERE` で使うと遅くなります。常に同じ結果になる数式は「決定的」でインデックス対象にできます。

> [!注意] クエリプランツールは SOQL 専用
>
> クエリ実行プランツールで評価できるのは **SOQL のみ**です。SOSL は評価できませんが、先頭ワイルドカードを避けるなどパフォーマンス配慮は必要です。

## 試験対策：押さえておきたい追加ポイント

> [!まとめ] 効率的なクエリの要点整理
>
> - 遅いクエリは**ガバナ制限**に引っかかる原因になる。
> - 速さの鍵は**インデックス**。`WHERE` でインデックス付き項目を使うと**セレクティブ**になる。
> - 自動インデックス項目：`Id` / `Name` / `OwnerId` / `CreatedDate` / `SystemModStamp` / `RecordType` / 主従・参照項目 / 一意項目 / 外部 ID 項目。
> - **セレクティブを壊す操作**：null 照会・否定演算子（`!=`, `NOT LIKE`, `EXCLUDES`）・先頭ワイルドカード（`'%...'`）・テキスト項目への比較演算子（`>`, `<`）。
> - `LastModifiedDate` より**インデックスの効く `SystemModStamp`**。
> - **クエリ実行プランツール**（開発者コンソールの設定で有効化）で `TableScan` か `Index` 利用かを確認。SOQL 専用。
> - **非決定性数式項目**はインデックス不可なので `WHERE` で避ける。
> - カスタムインデックスは Salesforce サポートに依頼できる。
