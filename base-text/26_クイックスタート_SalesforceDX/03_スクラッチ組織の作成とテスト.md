# スクラッチ組織の作成とテスト

## 学習の目的

この単元を完了すると、次のことができるようになります。

- Salesforce CLI でスクラッチ組織を作成し、開く。
- ローカルのソースメタデータをスクラッチ組織へリリース（デプロイ）する。
- 権限セットを割り当て、サンプルデータをインポートする。
- スクラッチ組織上でアプリケーションが正しく動くことを確認する。

> [!ポイント] この単元のゴール
>
> いよいよ DX の本番です。**スクラッチ組織を作る → コードをリリース → 権限セットを割り当て → データを入れる → 動作確認**という一連の流れ（スクラッチ組織のライフサイクル）を体験します。コマンドの順番と、それぞれが何をしているかをセットで押さえましょう。

## はじめに：このステップで行うこと

このステップでは、スクラッチ組織を作成し、権限セットを割り当て、サンプルデータをプッシュします。

> [!例] スクラッチ組織のライフサイクル（全体像）
>
> ```text
> ① 作成 create        ② 開く open          ③ リリース deploy
>   ┌─────────┐         ┌─────────┐          ┌──────────────┐
>   │ 空の組織を │  ──▶   │ ブラウザで │   ──▶   │ ローカルのコードを │
>   │ 発行する  │         │ 組織を開く │          │ 組織へ反映する   │
>   └─────────┘         └─────────┘          └──────────────┘
>        │                                            │
>        ▼                                            ▼
> ④ 権限付与 assign permset   ⑤ データ投入 import tree   ⑥ テスト test
>   ┌──────────────┐         ┌──────────────┐        ┌─────────────┐
>   │ アクセス権を   │  ──▶    │ サンプルデータを │  ──▶   │ アプリを開いて │
>   │ 割り当てる    │         │ 取り込む       │        │ 動作を確認する │
>   └──────────────┘         └──────────────┘        └─────────────┘
>        （用が済んだら）  ──▶  ⑦ 削除 delete scratch（割り当てを解放）
> ```

> [!注意] スクラッチ組織の作成数には上限がある
>
> Dev Hub のエディションによって、**1日に作成可能な数**と**一時点に有効化できる数**が決まります。上限に達したら `sf org delete scratch` で既存のスクラッチ組織を削除し、枠を解放します。

## スクラッチ組織を作成する

スクラッチ組織を作成してデフォルトに設定し、別名を付けます。

```bash
# スクラッチ組織を作成する
#   -d : この組織を以降のコマンドのデフォルト組織に設定する
#   -f : 組織の構成を定義した設定ファイルを指定する
#   -a : 別名（エイリアス）として dreamhouse-org を付ける
sf org create scratch -d -f config/project-scratch-def.json -a dreamhouse-org
```

> [!用語] `project-scratch-def.json`（スクラッチ組織定義ファイル）
>
> スクラッチ組織の「設計図」となる JSON 設定ファイル。組織のエディションや有効化する機能などを記述すると、その内容どおりの組織が発行されます。`config/` フォルダにあり、`-f` フラグで指定します。中身は次のようなイメージです。
>
> ```json
> {
>   "orgName": "DreamHouse Company",
>   "edition": "Developer",
>   "features": [],
>   "settings": {}
> }
> ```

通常 1 分以内に完了し、組織 ID とユーザー名が出力されます。パスワードが表示されないのは、Salesforce DX が**キャッシュされた認証トークン**を使うためです。

> [!用語] 認証トークン（authentication token）
>
> ログイン状態を表す「合言葉」。一度認証すると CLI がトークンを保持するため、コマンド実行のたびにパスワードを入力する必要がありません。スクラッチ組織でパスワードが表示されないのはこの仕組みのためです。

次のコマンドで、作成したスクラッチ組織を開きます。

```bash
# デフォルトのスクラッチ組織をブラウザーで開く
sf org open
```

> [!ポイント] `-d`（デフォルト指定）の効きどころ
>
> 作成時に `-d` を付けたため、認証トークンが記憶・保存され、`sf org open` をフラグなしで実行すると**デフォルト組織**が開きます。手動ログインやパスワードを思い出す手間が省けます。デフォルト組織は `sf config get target-org` で確認できます。

次に、前の単元でコピーした dreamhouse-lwc プロジェクトをスクラッチ組織にリリースします。

---

## スクラッチ組織にソースメタデータをリリースする

次のコマンドで dreamhouse-lwc プロジェクトをスクラッチ組織にリリース（デプロイ）します。

```bash
# ローカルのソース（メタデータ）をデフォルトのスクラッチ組織へリリースする
sf project deploy start
```

> [!用語] メタデータ（metadata）
>
> Salesforce における「設定や構成の情報」。Apex クラス、Lightning Web コンポーネント（LWC）、オブジェクト定義、タブ、ページレイアウトなどがすべてメタデータです。DX ではこれらをローカルのファイルとして扱い、組織へ**リリース（deploy）**します。

> [!用語] リリース／デプロイ（deploy）
>
> ローカルのメタデータ（コードや設定）を Salesforce 組織へ反映・配置すること。`sf project deploy start` で、プロジェクト内のソースがスクラッチ組織にまとめて送り込まれます。

しばらく時間がかかりますが、すべてのメタデータがリリースされ、ターミナルに正常にリリースされたリソースのリストが表示されます。

---

## DreamHouse アプリケーションに権限セットを割り当てる

DreamHouse アプリは権限セットでアクセス権を提供します。テスト前に権限セットを割り当てる必要があります。Salesforce CLI ではリモートで割り当てができます。

> [!用語] 権限セット（Permission Set）
>
> ユーザーにオブジェクトや項目・機能へのアクセス権を**追加で付与**する仕組み。プロファイルの基本権限に「上乗せ」する形で使い、必要なユーザーにだけ機能を開放できます。DreamHouse アプリの画面やデータを見るにはこの割り当てが必要です。

次のコマンドで権限セットを割り当てます。

```bash
# Dreamhouse 権限セットを現在のユーザーに割り当てる
#   -n : 割り当てる権限セットの名前（API 名）を指定する
sf org assign permset -n Dreamhouse
```

> [!注意] テスト前に権限セットの割り当てを忘れない
>
> 権限セットを割り当てないと、アプリのタブやデータが**表示されない**ことがあります。リリース直後に画面が空に見えたら、まず権限セットを割り当てたか確認しましょう。

## テストデータをインポートする

アプリのテストにはデータが必要です。dreamhouse-lwc に含まれるサンプルデータをスクラッチ組織にプッシュします。データのインポートには **sObject ツリー保存 API** を使用します。

> [!用語] sObject ツリー保存 API（SObject Tree Save API）
>
> 親子関係（ツリー構造）を持つレコードを、関係を保ったまままとめて組織へ登録できる API。たとえば「仲介業者（親）とその物件（子）」を、1回のインポートで整合性を保って投入できます。

次のコマンドでサンプルデータをインポートします。

```bash
# 計画ファイルに従ってサンプルデータをスクラッチ組織へインポートする
#   -p : データインポートの計画（plan）ファイルを指定する
sf data import tree -p data/sample-data-plan.json
```

> [!用語] データプランファイル（sample-data-plan.json）
>
> どのデータファイルを、どの順番で、どのオブジェクトへ取り込むかを定義した JSON ファイル。`-p` フラグで指定し、親レコードを先・子レコードを後に取り込むといった**インポートの段取り**が記述されています。

これでスクラッチ組織の設定が完了し、アプリをテストする準備ができました。

---

## スクラッチ組織でアプリケーションをテストする

プロジェクトとデータをプッシュしたら、すべてが期待どおり存在するか確認します。

> [!手順] スクラッチ組織で DreamHouse アプリを確認する
>
> 1. スクラッチ組織がまだ開いていなければ開く。
>
>    ```bash
>    # デフォルトのスクラッチ組織をブラウザーで開く
>    sf org open
>    ```
>
> 2. **アプリケーションランチャー**で **[Dreamhouse]** を見つけて開く。
> 3. **[Properties（不動産物件）]** タブで **12 個**の物件があることを確認する。
> 4. **[Brokers（仲介業者）]** タブで **8 人**の仲介業者がいることを確認する。

> [!ポイント] 件数で投入成功を確認できる
>
> 物件 **12 件**、仲介業者 **8 人**が表示されれば、メタデータのリリースとデータインポートが正しく完了した証拠です。件数が合わない場合は、リリース（`deploy`）・権限セット割り当て・データインポートのいずれかを見直しましょう。

お疲れさまでした。これで DreamHouse アプリの新機能開発を始める準備が整いました。引き続き VS Code など他の開発者ツールでの DX 活用や、継続的インテグレーション（CI）も学習しましょう。

> [!用語] 継続的インテグレーション（CI：Continuous Integration）
>
> コードの変更を頻繁に共有リポジトリへ統合し、その都度**自動でビルド・テスト**を実行する開発手法。問題を早期に発見でき、リリースを安定させられます。Salesforce DX は CI を組み込みやすい構成です。

| 順 | コマンド | 役割 |
| --- | --- | --- |
| 1 | `sf org create scratch -d -f config/project-scratch-def.json -a dreamhouse-org` | スクラッチ組織を作成（デフォルト・別名付き） |
| 2 | `sf org open` | スクラッチ組織をブラウザで開く |
| 3 | `sf project deploy start` | ローカルのメタデータを組織へリリース |
| 4 | `sf org assign permset -n Dreamhouse` | 権限セットを割り当て |

> [!まとめ] この単元のおさらい
>

> - コードは **`sf project deploy start`** でローカルから組織へリリースする。
> - アプリを使うには **権限セットの割り当て**（`sf org assign permset`）が必要。
> - テストデータは **`sf data import tree`** と**データプランファイル**で投入する。
> - 物件 12 件・仲介業者 8 人が表示されれば設定成功。不要になったら **`sf org delete scratch`** で削除。

## 試験対策：押さえておきたい追加ポイント

> [!ポイント] スクラッチ組織まわりの頻出ポイント
>
> - スクラッチ組織は **Dev Hub から作成**し、**最大30日間**有効。期限切れで使えなくなる。
> - 同時に有効化できる数・1日に作れる数は **Dev Hub のエディション**で決まる。上限到達時は `sf org delete scratch` で解放。
> - スクラッチ組織の構成は **`project-scratch-def.json`** で再現可能（ソース駆動の利点）。
> - `sf project deploy start` は**ローカル → 組織**へリリース。逆方向は `sf project retrieve start`。
> - 権限はプロファイルに加えて**権限セット**で上乗せ付与するのが現在の推奨。
> - データインポートは **sObject ツリー保存 API** を利用し、**プランファイル**で親子の取り込み順を制御する。
