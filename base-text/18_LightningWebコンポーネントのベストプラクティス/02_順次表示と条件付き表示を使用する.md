# 順次表示と条件付き表示を使用する

## 学習の目的

この単元を完了すると、次のことができるようになります。

- 順次表示（プログレッシブディスクロージャー）の考え方を説明する。
- 遅延インスタンス化（遅延読み込み）を実装する方法を理解する。
- `lwc:if|elseif|else` と CSS による条件付き表示の違いを説明する。

> [!ポイント] この単元のゴール
>
> 「**必要になるまで作らない・表示しない**」で初期読み込みを速くするのがテーマ。**遅延インスタンス化**（必要になるまで作らない）と**条件付き表示**（条件が満たされたら表示）の2本柱、そして条件付き表示の3つの方法（アプリケーションビルダー / `lwc:if` / CSS）の違いを押さえましょう。

---

## はじめに：なぜ順次表示が重要か

Lightning コンポーネントはページ読み込み時にインスタンス化され、全体の読み込み時間に影響します。そのためインタラクティブなデザインガイドラインでは順次表示が優先されます。順次表示は乱雑さ・認知負荷を減らしてユーザビリティを高め、高度な機能を遅延・副画面化することで学習を容易にしエラーを減らします。

コンポーネントの作成を遅らせる方法のうち、ここでは「遅延インスタンス化」（遅延読み込み）と「条件付き表示」を詳しく見ます。

> [!用語] 順次表示（プログレッシブディスクロージャー）
>
> 一度にすべてを見せず、**ユーザーが必要としたタイミングで段階的に情報や機能を提示する**デザイン手法。最初の画面をシンプルに保て、初期読み込みも速くなります。

> [!用語] インスタンス化（Instantiation）
>
> コンポーネント（設計図）から、実際に画面で動く「実体（インスタンス）」を生成すること。処理時間がかかるため、**不要なコンポーネントを最初から全部インスタンス化しない**ことがパフォーマンス向上につながります。

> [!例] 順次表示の身近な例
>
> 申し込みフォームで「詳細オプションを表示」をクリックして初めて追加入力欄が現れる――これが順次表示。基本項目だけを先に見せ、必要な人にだけ詳細を出すことで画面も処理も軽くなります。

## 遅延インスタンス化

遅延インスタンス化（遅延読み込み）は、オブジェクトやコンポーネントが初めて使用されるまで作成されないことです。Lightning Experience や異なるタブコンポーネントで実行できます。

> [!用語] 遅延インスタンス化（Lazy Instantiation / 遅延読み込み）
>
> コンポーネントを最初から作らず、**実際に使われる瞬間まで作成を先送り**すること。タブを開いた瞬間やボタンを押した瞬間に初めてインスタンス化され、最初の表示負荷を減らせます。

### Lightning Experience での遅延インスタンス化

Lightning アプリケーションビルダーでは、遅延インスタンス化される領域にコンポーネントを配置することで順次表示を宣言的に実装できます。

| 配置領域 | 読み込みのタイミング |
| --- | --- |
| **標準タブコンポーネント** | ユーザーがタブを選択したときにのみ読み込む |
| **Lightning コンポーネントアクション / クイックアクション** | ユーザーがボタンをクリックしたときにのみ読み込む |
| **ユーティリティバー** | ユーザーがクリックしたときにコンポーネントを読み込む |

> [!用語] Lightning アプリケーションビルダー（Lightning App Builder）
>
> コードを書かず、ドラッグ＆ドロップでコンポーネントを配置・設定できる宣言的（ノーコード）ツール。「タブ」「アクション」「ユーティリティバー」に置くだけで遅延読み込みが自動適用されます。

### 独自のコンポーネントでの遅延インスタンス化

デフォルトで遅延インスタンス化をサポートするタブコンポーネント（`<lightning-tabset>`、`<lightning-tab>` など）を活用します。

```html
<lightning-tabset>
  <lightning-tab label="Item One"> Content for tab 1 </lightning-tab>
  <template lwc:if={showTabTwo}>
    <lightning-tab label="Item Two"> Content for tab 2 </lightning-tab>
  </template>
</lightning-tabset>
```

> [!ポイント] デフォルトで遅延インスタンス化されるコンポーネント
>
> `lightning-tabset` と `lightning-tab` は**デフォルトで遅延インスタンス化をサポート**。選択されていないタブの中身は、開かれるまで作成されません。試験頻出。

> [!注意] コンソールのサブタブは遅延読み込みされない
>
> タブセットとアプリケーションビルダーのタブは遅延読み込みされますが、**Lightning コンソール内のタブはワークスペースとして読み込まれ、サブタブは遅延読み込みされません**。コンソールアプリでは挙動が異なります。

## 条件付き表示

条件付き表示は、状態や動作が一致したときにのみオブジェクトやコンポーネントが表示されることです。LWC を条件付き表示する方法は 3 つあります。

1. Lightning アプリケーションビルダーの動的コンポーネント表示
2. `lwc:if|elseif|else`
3. CSS

> [!用語] 条件付き表示（Conditional Rendering）
>
> 「ある条件が真（true）のときだけ要素を表示する」仕組み。「エラー時だけメッセージを出す」「金額が一定以上のときだけ案内を出す」といった制御に使います。

```text
   条件付き表示の3つの方法（上から順に検討する）

   ① アプリケーションビルダーの動的コンポーネント表示  … 宣言的（ノーコード）。最優先
            │  （ただしアプリケーションビルダー対応コンポーネントのみ）
            ▼
   ② lwc:if | elseif | else                         … すべてのコンポーネントで使える
            │  （条件が満たされるまで作成・表示を遅延）
            ▼
   ③ CSS（slds-show / slds-hide）                    … 事前に作成しておき表示だけ切替
              （初期読み込みは速くならないが、状態が保持される）
```

---

### Lightning アプリケーションビルダーの動的コンポーネント表示

宣言型で、アプリケーションビルダーに組み込まれています。検索条件とロジックをプロパティに追加し、コンポーネントが Lightning ページに表示されるタイミングを制御します。

> [!例] 動的コンポーネント表示の使いどころ
>
> 「商談の金額が 100 万ドル以上のとき」だけ商談ページに特別な案内（リッチテキスト）を出す――こうした表示制御を、コードを書かずアプリケーションビルダーの検索条件で実現できます。最も手軽なので可能なら最優先で検討します。

---

### `lwc:if|elseif|else`

開発者が `lwc:if|elseif|else` で UI の部分を遅延インスタンス化し、DOM 要素を条件付き表示できます。

```html
<template>
  <div lwc:if={something}>Conditional text displayed if something = true</div>
  <div lwc:else>Conditional text displayed if something = false</div>
</template>
```

`something` が true なら最初の `<div>` とその子が作成され、2 つ目は表示されません。false に変わると結果が入れ替わり、最初の `<div>` が破棄され 2 つ目が表示されます。

> [!用語] DOM（Document Object Model）
>
> ブラウザーが HTML を解釈して作る「画面要素の木構造（ツリー）」。`<div>` などはこの DOM 上のノードです。`lwc:if` が false になると該当 `<div>` は**DOM から破棄**され、true に戻ると作り直されます。

> [!注意] `lwc:if` は要素を破棄・再作成するので状態が消える
>
> `lwc:if` で false になった要素は DOM から**破棄**され、再び true になると作り直されるため、入力値などの**状態は失われ（リセットされ）ます**。状態を保持したいなら CSS 方式を検討します。

---

### CSS

表示を切り替える CSS スタイルを活用する方法です。

**テンプレート**

```html
<div id="error" class={className}>{error}</div>
```

**JavaScript**

```javascript
get className() {
  // isError が true なら表示用クラス、false なら非表示用クラスを返す
  return isError ? 'slds-show' : 'slds-hide';
}
```

`<div>` とその子は前もって作成・表示されますが、JavaScript が実行されるまでユーザーには見えません。CSS は事前作成のため他 2 方式のように初期読み込みは速くなりませんが、実行後は初期化・表示処理なしですぐ表示できます。また CSS はコンポーネントがアクティブのまま保持されるため**状態が維持されます**（`lwc:if` は破棄・再作成され状態が失われます）。

> [!用語] SLDS（Salesforce Lightning Design System）/ `slds-show`・`slds-hide`
>
> Salesforce 公式デザインシステムの CSS クラス。`slds-show` は表示、`slds-hide` は非表示。要素は DOM 上に残したまま見た目だけ切り替えます。

> [!ポイント] `lwc:if` と CSS の違いを表で整理
>
> | 比較項目 | `lwc:if|elseif|else` | CSS（`slds-show`/`slds-hide`） |
> | --- | --- | --- |
> | 要素の作成タイミング | 条件が true になってから作成 | 最初から作成済み |
> | 初期読み込みの速さ | 速くなる（作成を遅延） | 速くならない（事前作成） |
> | 状態の保持 | 破棄・再作成され**失われる** | アクティブのまま**保持される** |
> | 表示後の即応性 | 作成処理が入る | すでに作成済みで即表示 |
>
> 「**最初を軽くしたいなら `lwc:if`、状態を保ちたい・即時表示したいなら CSS**」と覚えましょう。

---

### 3つの方法を使う順番

提示された順で使用します。まず宣言型の動的コンポーネント表示（アプリケーションビルダー対応コンポーネントのみ）、次にすべてのコンポーネントで機能する `lwc:if|else`。どちらも条件が満たされるまで作成・表示を遅延し初期読み込みを短縮します。CSS は事前読み込みしておき条件成立時に表示したいときに使います。

> [!ポイント] 検討順は「宣言型 → `lwc:if` → CSS」
>
> 1. **アプリケーションビルダーの動的コンポーネント表示**（宣言型・ノーコード）をまず検討。
> 2. 設定できないなら **`lwc:if|else`**（すべてのコンポーネントで動作）。
> 3. 事前作成して即時表示・状態保持したいときだけ **CSS**。

## 試験対策：押さえておきたい追加ポイント

> [!ポイント] 順次表示・条件付き表示の頻出ポイント
>
> - **デフォルトで遅延インスタンス化**されるのは `lightning-tabset` と `lightning-tab`。
> - **Lightning コンソールのサブタブ**は遅延読み込みされない（例外として狙われやすい）。
> - 条件付き表示は現行の **`lwc:if|elseif|else`** を使う（古い `if:true|false` ではない）。
> - **`lwc:if` は破棄・再作成 → 状態が失われる**。**CSS は事前作成・状態保持**だが初期読み込みは速くならない。
> - 検討順は「**宣言型 → `lwc:if` → CSS**」。

> [!まとめ] この単元のまとめ
>
> - **順次表示**は、必要になるまで作らない・見せないことで初期読み込みを速くしユーザビリティを高める手法。
> - **遅延インスタンス化**は `lightning-tabset` / `lightning-tab` がデフォルトでサポート（コンソールのサブタブは対象外）。
> - **条件付き表示**には 3 つの方法（アプリケーションビルダー / `lwc:if` / CSS）があり、この順で検討する。
> - **`lwc:if` は状態が消える、CSS は状態が残る**――この違いが最重要。
>
> 次はパフォーマンス改善に役立つ他のオプションを学習しましょう。
