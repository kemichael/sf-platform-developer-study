# 組織への変更を計画する

## 学習の目的

この単元を完了すると、次のことができるようになります。

- 組織開発モデルとは何か、どのような課題を解決するのかを説明する。
- アプリケーション開発ライフサイクルの各フェーズと、対応するサンドボックスの種類を説明する。
- 主なツール（Salesforce DX プロジェクト、Salesforce CLI、VS Code 向け Salesforce 拡張機能、ソース制御）の役割を説明する。
- 変更管理に使うメカニズム（変更リスト、リリース実行リスト、プロジェクト管理ツール）を説明する。

> [!ポイント] この単元のゴール
>
> 「**本番組織で直接カスタマイズする**やり方から、**ソース制御リポジトリを中心に据えたチーム開発**へ移行する」のが組織開発モデルです。「変更をどこに保存するか（ソース制御）」「どのサンドボックスを何に使うか」「どのツールが何をするか」の3点が試験対策の要です。

---

## ストーリー：システム管理者 Calvin の課題

移住サービス企業 Zephyrus Relocation Services のシステム管理者 Calvin Green は、本番組織の設定画面で直接 Salesforce をカスタマイズしてきました。チームが成長し、複数メンバーが同時に開発するようになると、次の課題に直面します。

- 全員が行った変更の追跡
- すべての開発環境・テスト環境の常時同期
- エラーや環境間の違いによる、変更セットの複数回の作り直し

リードデベロッパー Juan Garcia は、これらの問題の軽減に**組織開発モデル**が役立つと考えています。

> [!用語] 組織開発モデル（Org Development Model）
>
> 1つの「組織（本番組織）」をチームで開発・保守する手法。変更をソース制御リポジトリ（GitHub など）に保存し、サンドボックスとソース制御で開発→テスト→リリースを回します。「組織」を単位とするため**組織開発**と呼びます。再利用可能な部品（パッケージ）を単位とするモデルは**パッケージ開発**です。

> [!用語] 変更セット（Change Set）
>
> 設定画面（宣言型 UI）から、ある組織のメタデータを別の組織へコピーする Salesforce 標準のリリース機能。手軽な反面、**変更履歴が残らない・大規模化すると管理が大変・自動化しにくい**という弱点があります。組織開発モデルはこの弱点をソース制御で解決します。

## 組織開発に移行する

組織開発モデルは、1) より高い柔軟性と拡張性、2) 変更の追跡と管理の新しい方法、3) 異なるリリース方法を提供します。変更とプロジェクトファイルをソース制御リポジトリに保存することで、環境を常時同期させる問題が軽減されます。各リリースの変更を外部化すれば、リポジトリが提供物を反映し、環境の違いに影響されなくなります。

VS Code 向け Salesforce 拡張機能で開発環境からメタデータを取得し、ソース制御に保存します。保存後はテストと本番リリースの自動化プロセスを作成でき、複数回のテストとインテグレーションを経て本番にリリースします。

> [!用語] ソース制御リポジトリ／バージョン管理
>
> ファイルの変更履歴を記録・管理する仕組み。**Git** が代表的なツールで、**GitHub** は Git リポジトリをクラウドでホストするサービス。「いつ・だれが・何を変更したか」が残り、過去に戻したり複数人で安全に編集したりできます。

> [!用語] 外部化（Externalize）
>
> 組織の中（本番・サンドボックス）にしかない変更を、外部のソース制御リポジトリへ取り出して保存すること。変更が「一元的な情報源（Single Source of Truth）」に集まり、どの環境が正しいかで迷わなくなります。

> [!用語] メタデータ（Metadata）
>
> 「データそのもの」ではなく「組織の構成・設定を定義する情報」。カスタムオブジェクト、項目、Apex クラス、トリガー、プロファイルなどがメタデータです。これをファイルとして取得し、ソース制御で管理します。

> [!例] 組織開発モデルへの移行で何が変わるか
>
> - **これまで**：本番組織の設定画面で直接ダッシュボードを作成 → 記録なし、環境ごとにバラバラ。
> - **これから**：VS Code でローカル取得 → GitHub にコミット → サンドボックスでテスト → 本番リリース。すべての変更が GitHub に記録され、巻き戻しもチーム共有も自在。

## リリース環境を準備する（サンドボックスの使い分け）

チームはアプリケーションライフサイクルの各ステップでサンドボックスを使います。

> [!用語] サンドボックス（Sandbox）
>
> 本番組織のコピーとして作られる開発・テスト専用の組織。本番に影響を与えず開発・検証できます。コピー内容（メタデータのみ／一部データ／全データ）と容量・更新間隔で種類が分かれ、用途に応じて使い分けます。

- **開発とテスト**：各メンバーが独自の **Developer Sandbox**（本番データなし）で担当のカスタマイズを作成。
- **リリースのビルド**：各自のカスタマイズを共有の **Developer Pro Sandbox**（本番データなし／テストデータ投入可）に統合。
- **リリースのテスト**：**Partial Sandbox** で本番組織のレプリカを作り、ユーザー受け入れテストを実施。
- **リリース**：本番リリース時、**Full Sandbox**（本番データのコピーあり）で本番に影響を与えずユーザーをトレーニング。

> [!用語] ユーザー受け入れテスト（UAT：User Acceptance Test）
>
> 実際の利用者（営業チーム）が「自分たちの業務要件を満たしているか」を確認する最終段階のテスト。開発者の動作確認とは別に、**ビジネス側の視点**で「使えるか」を判断します。

**どのフェーズでどのサンドボックスを使うかが試験で問われます。**

| サンドボックス種別 | コピーされる内容 | 本番データ | 主な用途（フェーズ） | このストーリーでの担当 |
| --- | --- | --- | --- | --- |
| **Developer** | メタデータのみ | 含まれない | 個人の開発・単体テスト | 各メンバーが個別に所有 |
| **Developer Pro** | メタデータのみ（容量大） | 含まれない（テストデータは投入可） | 変更の統合・統合テスト | チーム共有 |
| **Partial Copy（Partial）** | メタデータ＋本番データの一部サンプル | 一部 | ユーザー受け入れテスト（UAT） | UAT 用レプリカ |
| **Full** | メタデータ＋本番データ全部 | すべて | ステージング・回帰テスト・本番リリースのリハーサル | 最終検証・トレーニング |

> [!ポイント] サンドボックスはデータ量とフェーズで覚える
>
> 「**Developer → Developer Pro → Partial → Full**」の順に**含まれるデータが増え、本番に近づく**。Developer 系は**本番データを含まない**、Partial は**一部サンプル**、Full は**本番データ全部のコピー**で本番リリースのリハーサルに使います。

```text
 ┌──────────────────┐   ┌──────────────────────┐   ┌──────────────────┐   ┌──────────────┐
 │ Developer        │   │ Developer Pro        │   │ Full Sandbox     │   │ 本番組織      │
 │ Sandbox          │──▶│ Sandbox              │──▶│ （ステージング） │──▶│ Production    │
 │ 開発してテスト   │   │ 各自の変更を統合     │   │ テストして検証   │   │ リリース      │
 └────────┬─────────┘   └──────────┬───────────┘   └────────┬─────────┘   └──────┬───────┘
          │                        │                        │                    │
          ▼                        ▼                        ▼                    ▼
   ┌────────────────────────────────────────────────────────────────────────────────┐
   │          ソース制御リポジトリ（GitHub）── すべての変更をここに保存             │
   └────────────────────────────────────────────────────────────────────────────────┘
```

> [!注意] 本ストーリーではサンドボックスの代わりに Developer Edition / Playground を使う
>
> 実際に手を動かす場合、各サンドボックスの代わりに **Developer Edition 組織**または **Trailhead Playground** を使ってかまいません。サンドボックスは「本当の」開発作業のために残しておきましょう。

## プロ並みに変更を管理する（使用するツール）

組織開発モデルでは、使い慣れたツールに加えていくつか新しいツールを使い、特定のリリースフェーズで生産性を高めます。

### Salesforce DX プロジェクト

DX プロジェクトには変更を構成するソースとファイル、設定ファイル `sfdx-project.json` が含まれます。`sfdx-project.json` にはプロジェクト情報があり、多くの開発タスクに Salesforce DX ツールを利用できるようになります。

> [!用語] Salesforce DX（Salesforce Developer Experience）
>
> ローカル（自分の PC）でメタデータをソースファイルとして扱い、CLI やソース制御と組み合わせて開発するためのモダンな手法・ツール群の総称。作業単位が **DX プロジェクト**で、`sfdx-project.json` を核にした決まったフォルダー構造を持ちます。

```text
language-courses/                ← DX プロジェクトのルート
├── .sfdx/                       ← CLI が使う内部ファイル
├── .vscode/                     ← VS Code 用の設定
├── config/                      ← スクラッチ組織などの設定
├── force-app/                   ← ソース形式の変更が入る（メインの作業場所）
│   └── main/default/
│       ├── objects/             ← カスタムオブジェクト・項目
│       └── triggers/            ← Apex トリガー
├── manifest/                    ← package.xml が入る
├── .forceignore                 ← 取得・リリース対象から除外するファイルの指定
└── sfdx-project.json            ← プロジェクト用の設定ファイル
```

> [!用語] ソース形式（Source Format）
>
> メタデータを、人が読みやすく Git で差分管理しやすいよう複数のフォルダー・ファイルに分割して保存する形式。`force-app` 以下に配置されます。従来のメタデータ API 形式（1つの大きな zip）よりチーム開発向きです。

### リリースアーティファクト（マニフェストファイル）

変更をテストした後、Juan はリリースされるコンポーネントを列挙したマニフェストファイル（`package.xml`）を作成します。これを使ってまず各サンドボックスへ、最後に本番組織へリリースします。

> [!用語] マニフェストファイル（package.xml）／リリースアーティファクト
>
> 「どのメタデータコンポーネントをリリース（または取得）するか」を**一覧で記述した XML ファイル**。実際のコードやオブジェクトの中身は含まず、**名前のリストだけ**を持ちます。これ自体が成果物（アーティファクト）として、各サンドボックス→本番へ順番にリリースに使われます。

### ソース制御システム

すべての変更はマージされ、DX プロジェクトを含むソース制御システムに保存されます。利点は次のとおりです。

- リアルタイムのコラボレーションで効率が向上する。
- 上書きや作業損失を心配せず、チームで同じファイルを同時に作業できる。
- 変更履歴で、だれがどの変更を行ったかわかる。
- どのファイルも以前のバージョンに戻せる。
- コミットの説明で作業の過去の状況を示せる。

> [!用語] コミット（Commit）／マージ（Merge）
>
> **コミット**は、変更内容に説明（コミットメッセージ）を付けてリポジトリに記録すること（作業のセーブポイント）。**マージ**は、別々に進めた変更（ブランチ）を1つに統合すること。同じファイルを扱うときの競合はマージで解決します。

### Salesforce CLI

組織開発ライフサイクルの全フェーズで使えるコマンドラインインターフェース。開発・テスト・自動化を単一のインターフェースで行えます。主な操作：

- Sandbox の承認（ヘッドレス／Web フロー）
- DX プロジェクトの作成と管理
- テストデータのインポート／エクスポート
- メタデータの取得とリリース
- テストの実行と自動化

> [!用語] Salesforce CLI ／ `sf` コマンド
>
> ターミナルから Salesforce を操作するツール。現在は `sf` で始まるコマンドを使います（例：`sf project deploy start`）。クリックの代わりにコマンドで操作するため、**手順を自動化（スクリプト化）** でき、CI/CD に欠かせません。

> [!例] よく使う sf コマンド
>
> - `sf project retrieve start` … 組織からローカルへメタデータを**取得**
> - `sf project deploy start` … ローカルから組織へメタデータを**リリース（デプロイ）**
> - `sf project generate manifest` … `package.xml` を**生成**
> - `sf org open` … 対象の組織を**ブラウザで開く**

### VS Code 向け Salesforce 拡張機能

Salesforce CLI と VS Code を基盤とした、Lightning Platform 開発向けの統合開発環境。CLI コマンドをコマンドパレットやターミナルから直接実行できます。**Salesforce Extension Pack** をインストールすると以下を使えます。

| 拡張機能 | 役割 |
| --- | --- |
| **開発者向け Einstein** | 自然言語の指示でコード候補を生成。 |
| **Salesforce CLI インテグレーション** | Salesforce CLI と連携してコア機能を提供。 |
| **Apex** | Apex 言語サーバーで構文強調・コード補完などを提供。 |
| **Apex Replay Debugger** | Apex デバッグログから Apex 実行を再生。 |
| **Lightning Web コンポーネント** | LWC をサポート。 |

> [!用語] 統合開発環境（IDE）／コマンドパレット
>
> **IDE** はコードの編集・補完・デバッグ・実行を1つにまとめた開発ツール。**VS Code** が代表。**コマンドパレット**は VS Code でコマンドを名前検索して実行する入力欄で、`Ctrl+Shift+P`（Mac は `Cmd+Shift+P`）で開きます。「SFDX:」と打つと Salesforce 関連コマンドが一覧表示されます。

> [!ポイント] CLI が使えない環境では Salesforce Code Builder
>
> セキュリティ制限で CLI や VS Code をインストールできない、またはクラウドのみで作業したい場合は、**Salesforce Code Builder** を検討します。VS Code・Salesforce 拡張機能・CLI のすべての機能を **Web ブラウザー**で使える Web ベースの IDE です。

## 変更管理のメカニズム

| ツール | 説明 |
| --- | --- |
| **変更リスト（Change List）** | 自分の開発組織で行った変更を追跡するシンプルなリスト・表・スプレッドシート。外部化すべき項目を把握できる。 |
| **リリース実行リスト（Release Runbook）** | リリース前後に必要な手動変更（プロファイルや権限セットの割り当てなど、リリースできないメタデータ）をリストする。 |
| **プロジェクト管理ツール** | アジャイルアクセラレーターや Jira など。ビジネス要件・今後の作業・バグの追跡、スプリント管理、バックロググルーミングに役立つ。 |

> [!注意] 「リリースできないメタデータ」がある
>
> プロファイルや権限セットの割り当てなど、一部の設定は**自動リリースに含められない**ことがあります。これを**リリース実行リスト**に書き出しておかないと、本番リリース後に「営業チームが新しい項目を見られない」といった抜け漏れが起きます。後の単元で実際にこのリストを使います。

> [!用語] バックログ／スプリント（アジャイル開発）
>
> **バックログ**は「これからやる作業項目を優先度順に並べたリスト」。**スプリント**は「短い期間（例：2週間）で区切った開発サイクル」。バックログから項目を選び、スプリントごとに少しずつ完成させます。

### ツールをインストールする

Juan と Ella は VS Code 向け Salesforce 拡張機能を使い、ソース制御に GitHub を使います。

> [!手順] 開発に必要なツールを準備する
>
> 1. Salesforce CLI をインストールする。
> 2. VS Code と Salesforce 拡張機能をインストールする。
> 3. Git をインストールする。
> 4. Git を設定する。
> 5. GitHub アカウントを作成する。

> [!ポイント] Salesforce CLI オートコンプリート
>
> オートコンプリート機能を入れると、CLI コマンドやフラグの一部を入力して `Tab` キーで自動補完されます。ターミナルでの作業が大幅に楽になります。

## 試験対策：押さえておきたい追加ポイント

> [!ポイント] 組織開発モデル vs 変更セット vs パッケージ開発
>
> | 観点 | 変更セット | 組織開発モデル | パッケージ開発モデル |
> | --- | --- | --- | --- |
> | 開発の単位 | 組織（メタデータの塊） | 組織 | パッケージ（再利用可能な部品） |
> | 変更の保存先 | 組織内のみ | **ソース制御リポジトリ** | ソース制御リポジトリ |
> | 変更履歴 | 残らない | **残る（Git）** | 残る（Git） |
> | リリース方法 | UI で組織間コピー | CLI／メタデータ API でデプロイ | パッケージのインストール |
> | 自動化 | しにくい | **しやすい** | しやすい |
> | 向いているケース | 小規模・単発の変更 | 既存組織を継続改修するチーム | 複数組織へ配布・AppExchange |

> [!ポイント] この単元の頻出ポイント
>
> - 核心は「**変更をソース制御リポジトリで外部化（一元管理）する**」こと。
> - ソース制御の利点は「**複数人が同じファイルセットで同時に作業できる**」「変更履歴が残る」「巻き戻せる」。
> - サンドボックスは **Developer / Developer Pro（本番データなし）→ Partial（一部データ）→ Full（全データ）** の順でフェーズと用途が決まる。
> - `sfdx-project.json` は **DX プロジェクトの設定ファイル**、`package.xml` は **リリース対象を列挙したマニフェスト**。混同しない。

## テスト

この単元を完了するには、テストのすべての質問に正しく解答する必要があります（+100 ポイント）。

**1. 組織開発モデルについて、正しい説明はどれですか?**

- A. パッケージを使用して、環境間でメタデータを移動する。
- B. すべてのカスタマイズを Sandbox で作成することが必要である。
- C. ソース制御リポジトリで変更を外部化して、環境を常時同期する。
- D. ソース制御リポジトリを使用するため、開発者は手動で変更を追跡する必要がない。

**2. 組織開発モデルでソース制御システムを使用する利点の 1 つはどれですか?**

- A. 開発者が同じファイルセットで同時に作業できる。
- B. Sandbox で行われた変更を追跡する必要がなくなる。
- C. 特定の時点における本番環境のスナップショットが提供される。
- D. 他のユーザーが作成した変更とメタデータをデベロッパーが表示できないようにする。

> [!まとめ] この単元のまとめ
>
> - **組織開発モデル**は、本番組織を直接いじる方式から、**ソース制御リポジトリを中心としたチーム開発**へ移行する手法。「変更追跡」「環境同期」「変更セットの作り直し」を解決する。
