# 変更をテストしてリリースする

## 学習の目的

この単元を完了すると、次のことができるようになります。

- リリースアーティファクト（マニフェストファイル `package.xml`）を作成する。
- 組織への変更のリリースに使用されるコマンドを説明する。
- クイックリリースを使用してリリースを高速化する方法を説明する。

> [!ポイント] この単元のゴール
>
> 開発した変更を「**Developer Pro Sandbox（統合）→ Partial Sandbox（UAT）→ Full Sandbox（ステージング）→ 本番組織**」へ順番に進めてリリースします。`package.xml`（マニフェスト）の役割、`deploy validate` と `deploy quick`（クイックリリース）の使い分けが試験の要です。

---

## このストーリーの全体像（リリースの流れ）

開発タスクを完了した Juan と Ella は、変更を統合・テストしながら本番へ進めます。Juan は最終的なリリースアーティファクト（`package.xml`）を作成し、各サンドボックスでテストしてから本番組織にリリースします。

```text
 GitHub                Developer Pro          Partial              Full                 本番組織
 リポジトリ            Sandbox                Sandbox              Sandbox              Production
 （一元的な情報源）  ─▶ 変更を統合・    ─▶  UAT（一部の    ─▶  ステージング・  ─▶  検証 → クイック
   変更をすべて pull     リリーステスト      本番データ）       回帰テスト・         リリース
                                                                検証→クイックリリース
        │                   ▲
        │  package.xml を生成 │
        └────────────────────┘
```

> [!用語] リリースアーティファクト／マニフェストファイル（package.xml）
>
> リリースする対象を定めた成果物。リリースするメタデータコンポーネントを**名前で列挙した XML ファイル `package.xml`** を指します。実際のコードやオブジェクトの中身は含まず、「何をリリースするかのリスト」。同じ `package.xml` を各サンドボックス → 本番へ使い回します。

## 変更をリポジトリから取得する

Juan は、このリリースのすべての変更が GitHub リポジトリにそろったことを認識し、最新の変更を pull します。

> [!手順] リポジトリから最新の変更を取得（pull）する
>
> 1. VS Code で **[Source Control（ソース制御）]** アイコン（1）をクリックする。
> 2. **[追加アクション]** アイコン（2）をクリックし、**[Pull from（取得元）]**（3）を選択する。
> 3. オリジンを選択する。

> [!用語] プル（Pull）
>
> リモート（GitHub）の最新の変更を、自分のローカルに取り込んで同期する操作。リリース前に pull することで、**チーム全員の変更がそろった状態**でリリース作業を始められます。

> [!例] なぜリリース前にリポジトリから pull するのか
>
> Ella のオブジェクト・項目と Juan のトリガー・テストは、別々のブランチで作られてマージ済みです。リリース担当の Juan は、まず GitHub から pull して**全員分の変更が1つにそろったローカル**を用意します。これがリリースの出発点です。

## Developer Pro Sandbox を承認する

> [!注意] 同じ手順を試す場合
>
> Developer Edition 組織または Trailhead Playground にサインアップして、Developer Pro Sandbox の代わりに使用してください。

> [!手順] Developer Pro Sandbox を承認する
>
> 1. VS Code で **[SFDX: Authorize an Org（組織を承認）]** を選択する。
> 2. ログイン URL（`test.salesforce.com`）に **[Sandbox]** を選択する。
> 3. Sandbox の別名（`dev_pro_sandbox` など）を入力する。
> 4. Sandbox のユーザー名とパスワードでログインする。

> [!用語] Developer Pro Sandbox
>
> 本番データを含まないサンドボックスのうち、容量が大きくテストデータを取り込めるもの。**各メンバーの変更を1か所に統合し、統合テストを行う**フェーズで使います。

## リリースアーティファクトを構築する

Juan は変更を Developer Pro Sandbox にリリースできるよう、リリースするメタデータを列挙したマニフェストファイル（`package.xml`）を作成します。

> [!手順] package.xml を生成して Developer Pro Sandbox でテストする
>
> 1. Salesforce DX プロジェクトディレクトリが表示されていることを確認する。
> 2. `project generate manifest` コマンドのヘルプを表示する。

```bash
# コマンドの使い方（書式・フラグ）を確認する
sf project generate manifest --help
```

Juan は新しいメタデータコンポーネントのみをリリースしたいため、`--metadata` フラグが必要だと判断します。

> [!ポイント] 迷ったら `--help`
>
> どの CLI コマンドでも末尾に `--help` を付けると**書式・使えるフラグ・例**が表示されます。コマンドを丸暗記しなくても `--help` で必要なフラグを確認できる、というのが Salesforce CLI の使い方の基本です。

> [!手順] 続き
>
> 3. `project generate manifest` を実行し、リリースするコンポーネント（新しい Language Course Instructors オブジェクトや変更リストのその他の変更）を指定する。

```bash
# 変更リストの全コンポーネントを指定して package.xml を生成する
sf project generate manifest --metadata CustomObject:Language_Course_Instructor__c --metadata CustomField:Language_Course__c.Course_Instructor__c --metadata ApexTrigger:LanguageCourseTrigger --metadata ApexClass:TestLanguageCourseTrigger
```

現在のディレクトリに `package.xml` が生成されます。

> [!手順] 続き
>
> 4. Developer Pro Sandbox で `package.xml` をテストし、すべてのコンポーネントがリリースされることを確認する。

```bash
# 生成した package.xml を使って Developer Pro Sandbox にリリースする
sf project deploy start --manifest package.xml --target-org dev-pro-sandbox
```

> [!注意] package.xml は「リスト」、中身はローカルのソース
>
> `package.xml` には**コンポーネントの名前だけ**が並びます。実際にリリースされる中身（Apex コードやオブジェクト定義）は、**コマンド実行時にローカルにあるソースファイル**です。そのため、**どのブランチで・どの状態のファイルに対して** deploy するかが結果を左右します。意図しないブランチからリリースしないよう注意しましょう。

> [!用語] 破壊的な変更（Destructive Changes）
>
> マニフェストファイルでリリース中にコンポーネントを**削除**することもできます。削除対象を列挙する「破壊的な変更のマニフェスト」は `project generate manifest` の `--type` フラグで生成し、`project deploy start|validate` の削除フラグ（`--post-destructive-changes` など）と併せて指定します。

> [!注意] テストクラスは別モジュール扱い
>
> 1人で実行している場合、ジャーニーはここで終わりです。次のステップで必要な `TestLanguageCourseTrigger` テストクラスは、このモジュールでは扱いません。

## テスト（Partial）Sandbox でリリースアーティファクトをテストする

Juan は先ほどと同じ `project deploy start` で変更を Partial Sandbox にリリースします。

> [!用語] Partial Sandbox（Partial Copy Sandbox）
>
> メタデータに加えて**本番データの一部サンプル**を含むサンドボックス。本番に近いデータで**ユーザー受け入れテスト（UAT）** を行うのに適しています。

> [!手順] Partial Sandbox にリリースして UAT を行う
>
> 1. Partial Sandbox を承認する。
> 2. Salesforce DX プロジェクトディレクトリが表示されていることを確認する。
> 3. deploy コマンドのヘルプを表示する。

```bash
sf project deploy start --help
```

`package.xml` を使ってリリースするには `--manifest` を使います。

> [!手順] 続き
>
> 4. 本番組織へのリリースを模倣する deploy コマンドを実行する。

```bash
# 指定したテストだけを実行しつつ、package.xml の内容を Partial Sandbox にリリースする
sf project deploy start --manifest package.xml --target-org partial-sandbox --test-level RunSpecifiedTests --tests TestLanguageCourseTrigger
```

> [!手順] 続き
>
> 5. 必要に応じて Selenium テストなどの UI テストを実行する。
> 6. Sandbox を開く。

```bash
sf org open --target-org partial-sandbox
```

> [!手順] 続き
>
> 7. ユーザー受け入れテストを実施する。

このフェーズでは、リリースする変更に関連するテストのみを実行します。

> [!用語] テストレベル（`--test-level`）
>
> リリース時にどの Apex テストを実行するかの指定。
>
> | 値 | 実行されるテスト |
> | --- | --- |
> | `NoTestRun` | テストを実行しない（本番組織では基本的に不可） |
> | `RunSpecifiedTests` | `--tests` で指定したテストのみ |
> | `RunLocalTests` | 管理パッケージ以外の、組織内のすべてのローカルテスト |
> | `RunAllTestsInOrg` | 管理パッケージを含むすべてのテスト |
>
> UAT 段階では、関連テストだけを実行する `RunSpecifiedTests` を使います。

## ステージング（Full）Sandbox でリリースアーティファクトをテストする

統合テストで変更が不要だった場合、次に Full Sandbox でステージングし、回帰テストを実行して本番リリースを模倣します。エラーがなかったので、テスト時と同じ `package.xml`・同じブランチからリリースします（エラーがあれば修正してやり直す）。

> [!用語] Full Sandbox／回帰テスト（Regression Test）
>
> **Full Sandbox** は、メタデータと**本番データ全部のコピー**を含む、本番に最も近いサンドボックス。**回帰テスト**は「新しい変更が既存の機能を壊していないか」を確認するテスト。本番に最も近い環境で本番リリースのリハーサルを行います。

回帰分析は、まず**検証リリース**（組織に何も保存せず、テスト結果・連動関係・無効メタデータの有無を確認）で行います。すべての回帰テストを実行した後、**クイックリリース**で本番リリースの手順を完全に模倣します。ステージングで適切に検証することで、本番リリース時にユーザーアクセスをブロックするメンテナンス時間を短縮できます。

> [!用語] 検証リリース（validate）とクイックリリース（quick）
>
> - **検証リリース（`sf project deploy validate`）**：実際には**組織に何も保存せず**、テストの実行・連動チェック・無効メタデータの有無だけを確認する「リハーサル」。成功すると**ジョブ ID** が返る。
> - **クイックリリース（`sf project deploy quick`）**：直前の検証で実行済みのテストを**再実行せず**、検証済みの内容を一気に反映する。`--job-id` に検証時のジョブ ID を渡す。
>
> 「先に validate でテストを通し、本番では quick で素早く反映する」ことで、ユーザーへの影響時間を最小化できます。

> [!手順] Full Sandbox で検証 → クイックリリース
>
> 1. Full Sandbox を承認する。
> 2. 何も保存せず、すべてのローカル（回帰）テストを実行してリリースを検証する。

```bash
# 何も保存せず、ローカルテスト全実行でリリースを検証する（戻り値としてジョブ ID が返る）
sf project deploy validate --manifest package.xml --target-org full-sandbox --test-level RunLocalTests
```

検証成功は、すべての Apex テストが合格し、リリースされるコードの 75% 以上がテスト対象であることを意味します。

> [!手順] 続き
>
> 3. 前の手順で返されたジョブ ID を使い、Full Sandbox でクイックリリースを実行する（本番組織で行うことを模倣）。

```bash
# 検証で得たジョブ ID を使い、テストを再実行せずに素早くリリースする
sf project deploy quick --job-id <jobID> --target-org full-sandbox
```

> [!ポイント] 検証リリースの 75% ルール
>
> 検証（または本番リリース）が成功するには、**すべての Apex テストが合格**し、かつ**リリース対象のコードの 75% 以上**がテストでカバーされている必要があります。トリガーにテストクラスを必ず用意するのはこのためです。

## 本番組織にリリースする

Full Sandbox ですべてのテストに合格したため、本番組織にリリースする準備が整いました。Juan はリリース実行リストでリリース前タスクがないことを確認します。検証リリース後、組織に大きな変更を加えなければ 10 日間かけて本番へのクイックリリースを実施できます。Juan は、トラブルシューティングできるよう検証を営業時間中に実行し、影響を最小化するためクイックリリースをその日の夜に行います。

> [!ポイント] 検証リリースの「10 日間」ルール
>
> 検証リリースに成功すると、その結果は**10 日間有効**です。この期間内であれば、組織に大きな変更を加えていない限り、テストを再実行せず**クイックリリース**で本番反映できます。「**昼に検証 → 夜にクイックリリース**」とすればユーザーへの影響を最小化できます。

> [!手順] 本番組織への検証 → クイックリリース
>
> 1. 本番組織に対する承認を行う。
> 2. `project deploy validate` でクイックリリースの検証と設定を行う。

```bash
# 本番組織に対して、保存せずに検証だけ行う（ジョブ ID が返る）
sf project deploy validate --manifest package.xml --target-org production-org --test-level RunLocalTests
```

> [!手順] 続き
>
> 3. 返されたジョブ ID でクイックリリースを実行する。

```bash
# 検証済みのジョブ ID を使って本番組織へ素早くリリースする
sf project deploy quick --job-id <jobID> --target-org production-org
```

> [!手順] 続き
>
> 4. 本番組織を開いて、変更が表示されていることを確認する。

> [!注意] 本番リリースのテストレベル
>
> 本番組織へのリリースでは原則 `NoTestRun` は使えません。`RunLocalTests`（管理パッケージ以外の全テスト）か `RunSpecifiedTests`（指定テスト）でテストを実行し、カバー率 75% 以上をクリアする必要があります。ここでは安全側で `RunLocalTests` を使っています。

## リリース実行リストに記載されているリリース後タスクを実施する

Juan はリリース実行リストを参照し、本番組織で実行するリリース後タスクを確認します。

| フェーズ（前/後） | エンティティ/コンポーネント | メモ | 手順 |
| --- | --- | --- | --- |
| リリース前 | なし | 必要なタスクはありません | なし |
| リリース後 | セールスプロファイル | 営業チームがカスタムオブジェクトとカスタム項目を表示できるよう権限を更新します。 | [設定] で営業チームのプロファイルを編集し、[Language Course Instructors] の「参照」アクセス権を付与します。 |

> [!注意] プロファイルの権限はリリース後に手動で付与する
>
> プロファイルへのアクセス権付与のような設定は自動リリースに含めにくいため、**リリース実行リスト**に「リリース後タスク」として記録します。これを忘れると、本番にオブジェクトはリリースされても**営業チームが項目を表示できない**事態になります。第1単元で学んだリリース実行リストがここで効いてきます。

## 別のロールアウトにも成功!

Calvin は本番組織でサニティチェックを行います。インストラクターをコースに追加し、通知メールが届くことを確認します。すべての変更がソース制御に反映されているため、リリースノートに記載する変更の最終リストも提供できます。

> [!用語] サニティチェック（Sanity Check）
>
> リリース直後に行う「最低限の動作確認」。すべてを網羅するテストではなく、「主要な機能が壊れていないか」をざっと確かめること。ここでは「コースに講師を追加 → 通知メールが届く」を確認しています。

> [!例] ソース制御がもたらした最終的なメリット
>
> すべての変更が GitHub に残るため、「このリリースで何が変わったか」を**リリースノート**として自動的に洗い出せます。誰がいつ何を変えたかが追跡でき、問題があれば過去のバージョンに戻せる——これが組織開発モデルの一番の収穫です。

## 試験対策：押さえておきたい追加ポイント

> [!ポイント] リリース系コマンドのまとめ
>
> | コマンド | 役割 | 組織への保存 | 備考 |
> | --- | --- | --- | --- |
> | `sf project generate manifest` | `package.xml` を生成 | なし | `--metadata` で対象指定、`--type` で破壊的変更 |
> | `sf project deploy start` | リリース（テスト実行あり） | する | `--manifest` / `--source-dir` / `--metadata` |
> | `sf project deploy validate` | 検証リリース（リハーサル） | **しない** | 成功すると**ジョブ ID** を返す。結果は 10 日間有効 |
> | `sf project deploy quick` | クイックリリース | する | `--job-id` で検証結果を使い、テストを再実行しない |

> [!ポイント] この単元の頻出ポイント
>
> - **`package.xml`（マニフェスト）= リリースするメタデータを列挙した XML ファイル**。コードの中身や zip ではない。実際の中身はローカルのソースファイル。
> - **validate は組織に何も保存しない**（テスト結果と連動関係の検証のみ）。これが validate を使う最大のメリット。
> - **quick** は検証済みのジョブ ID を使い、**テストを再実行せず**素早くリリースする。検証は 10 日間有効。
> - リリースは **Developer Pro（統合）→ Partial（UAT）→ Full（ステージング/回帰）→ 本番** の順に段階的に行う。
> - プロファイルなど**リリースできない設定**は、リリース実行リストでリリース後タスクとして手動対応する。

## テスト

この単元を完了するには、テストのすべての質問に正しく解答する必要があります（+100 ポイント）。

**1. マニフェストファイル (package.xml) とは何ですか?**

- A. 組織にリリースするすべてのオブジェクトとコンポーネントを含む .zip ファイル。
- B. 組織にリリース（または組織から取得）するメタデータコンポーネントをリストした XML ファイル。
- C. 開発チームがこのリリースで行った変更のパッケージバージョン。
- D. Juan が本番環境にリリースする受信変更セット。

**2. project deploy validate コマンドを使用するメリットは何ですか?**

- A. 本番組織へのクイックリリースを設定できるため、ユーザーに対する中断を最小限に抑えることができる。
- B. 実際に組織に変更をコミットせずに、リリースで実行されるテストの結果を検証できる。
- C. 自分が CLI コマンドに精通していることを確認できる。
- D. A と B。

> [!まとめ] この単元のまとめ
>
> - リリースは「**リポジトリから pull → package.xml 生成 → Developer Pro（統合）→ Partial（UAT）→ Full（ステージング/回帰）→ 本番**」と段階的に進める。
> - **`package.xml`** は「リリース対象を列挙したマニフェスト」。同じファイルを各環境で使い回す。実際の中身はその時点のローカルソース。
> - **`deploy validate`** は組織に保存せずテスト結果と連動関係を検証（ジョブ ID を返す・10 日間有効）。**`deploy quick`** はその検証結果でテストを再実行せず素早くリリースし、本番のダウンタイムを最小化する。
> - リリース後は**リリース実行リスト**に従い、プロファイルの権限付与などの手動タスクを実施する。最後にサニティチェックで動作を確認する。

> [!注意] 日本語環境で受講する場合
>
> コマンドやフラグ、API 名はすべて英語です。手順を試す場合は `package.xml`・別名・メタデータ名などを正確に入力してください。サンドボックスの代わりに Developer Edition 組織または Trailhead Playground を使ってかまいません。
