# テストの復習

## 学習の目的

この単元を完了すると、次のことができるようになります。

- Apex テストのフレームワークと要件を説明する。
- テストデータの作成方法と考慮事項を説明する。
- テストの実行方法とコードカバレッジの要件を説明する。

> [!ポイント] この単元のゴール
>
> 試験直前の復習単元。「テスト、デバッグ、リリース」セクション（試験の **20%**）のうち**「テスト」トピック**を総ざらいする。中心は **フレームワーク・要件 / テストデータ / テストの実行 / 考慮事項**の4つ。特に **コードカバレッジ 75%**、**`@isTest`・`Test.startTest()`・`System.assert`** を確実に押さえる。

## 主なトピック

「テスト、デバッグ、リリース」セクション（試験の **20%**）の「テスト」トピックを確認する。対話型の練習問題とフラッシュカードが用意されている。

> [!用語] Apex テスト（単体テスト：Unit Test）
>
> 自分の Apex コードが正しく動くことをコードで検証する仕組み。テストメソッドで期待結果を**アサーション（assert）**で確認する。本番組織へのリリースにはテスト合格と一定のコードカバレッジが**必須**。

### この単元で復習する 4 トピック

| トピック | 押さえるポイント | 代表キーワード |
| --- | --- | --- |
| **フレームワークと要件** | テストクラスの書き方、合格条件、カバレッジ要件 | `@isTest`、75% カバレッジ |
| **テストデータ** | テスト内で作ったデータのみ参照が原則 | `@isTest(SeeAllData=false)`、`@TestSetup` |
| **テストの実行** | どこで・どう実行するか、ガバナ制限のリセット | 開発者コンソール、`Test.startTest()` |
| **テストの考慮事項** | バルク・例外・負のテストなどの観点 | `System.assertEquals`、バルクテスト |

## テストのフレームワークと要件

> [!用語] コードカバレッジ（Code Coverage）とアサーション
>
> - **コードカバレッジ**：テストで実行された Apex コード行の割合。本番リリースには**全体で 75% 以上**が必要。ただし目的は数字ではなく**意味のあるアサーションで動作を検証すること**。
> - **アサーション**：「期待値」と「実際値」を比較し一致しなければテストを失敗させる検証文。`System.assertEquals(期待値, 実際値, メッセージ)` や `System.assert(条件)`。アサーションのないテストはカバレッジは稼げても動作を保証しない。

> [!ポイント] テストの必須要件（最頻出）
>
> - テストメソッド・テストクラスには **`@isTest`** を付ける。
> - 本番リリースには **全体でコードカバレッジ 75% 以上**が必要。
> - 各 **Apex トリガー**は**最低 1% 以上**カバーが必要。
> - テストはすべて**成功（パス）**する必要がある。
> - テストコード自体は**カバレッジ計算に含まれない**。

> [!例] 基本的なテストクラスの形
>
> `@isTest` を付けたクラス・メソッドで対象を呼び出し、結果を `assert` で検証する。

```apex
@isTest
private class AccountServiceTest {

    @isTest
    static void testCreateAccount() {
        // 1. 準備：テスト用データを作成
        Account a = new Account(Name = 'テスト商事');

        // 2. 実行：対象コードを startTest/stopTest で囲む
        Test.startTest();
        insert a;
        Test.stopTest();

        // 3. 検証：アサーションで確認
        Account result = [SELECT Id, Name FROM Account WHERE Id = :a.Id];
        System.assertEquals('テスト商事', result.Name, '名前が一致しません');
    }
}
```

## テストデータ

> [!ポイント] テストデータの原則
>
> - テストは原則**テスト内で自分が作成したデータ**だけ参照できる（組織の既存データは見えない）。
> - 既定は **`@isTest(SeeAllData=false)`**。`SeeAllData=true` は組織データが見えるが**環境依存で壊れやすく非推奨**。
> - 共通データは **`@TestSetup`** メソッドで一度だけ作成し再利用できる。
> - **`Test.loadData()`** や**テストデータファクトリ**でデータ生成をまとめるとよい。

> [!注意] SeeAllData=true は避ける
>
> `SeeAllData=true` は組織の実データに依存し、データが変わるとテストが壊れる。原則 **`SeeAllData=false`（既定）** とし、必要なデータはテスト内で作成する。一部の標準オブジェクト（PricebookEntry など）は例外的に既存データ参照が必要。

> [!例] @TestSetup でデータを共通化
>
> `@TestSetup` で作ったデータは同じテストクラス内の各メソッドから利用でき、メソッドごとにロールバックされた状態から再利用される。

```apex
@isTest
private class OpportunityTest {

    @TestSetup
    static void makeData() {
        // クラス内の各テストメソッドで共有されるデータを作成
        Account a = new Account(Name = '共通取引先');
        insert a;
    }

    @isTest
    static void testSomething() {
        Account a = [SELECT Id FROM Account LIMIT 1]; // TestSetup のデータを参照
        System.assertNotEquals(null, a.Id);
    }
}
```

## テストの実行

> [!用語] Test.startTest() / Test.stopTest()
>
> テスト対象の処理を囲むメソッド。この間で実行されるコードには**新しいガバナ制限**が割り当てられる。さらに `Test.stopTest()` の時点で**非同期処理（`@future`・キュー可能・バッチ）が同期的に完了**する。非同期結果の検証に重要。

> [!ポイント] テストの実行場所と方法
>
> - **開発者コンソール**：個別／一括でテスト実行、カバレッジ確認。
> - **設定 → Apex テスト実行**：UI からまとめて実行。
> - **Salesforce CLI（`sf apex run test`）**：CI/CD やコマンドラインから実行。
> - リリース時は**デプロイ時に自動でテストが実行**される。

```text
   テスト実行のイメージ
   ┌──────────────────────────────┐
   │ 準備（Arrange）              │ テストデータを作成
   ├──────────────────────────────┤
   │ Test.startTest()             │ ← ここからガバナ制限がリセット
   │   実行（Act）                │ 対象コードを呼び出し
   │ Test.stopTest()              │ ← 非同期処理がここで完了
   ├──────────────────────────────┤
   │ 検証（Assert）              │ System.assertEquals で結果確認
   └──────────────────────────────┘
```

## テストの考慮事項

> [!用語] System.runAs()
>
> 指定したユーザーのコンテキスト（権限・共有設定）でテストコードを実行する仕組み。プロファイルや共有ルールによる挙動の違いの検証に使う。テスト専用で本番コードでは権限を昇格させない。

> [!ポイント] よいテストの観点（試験頻出）
>
> - **ポジティブ／ネガティブ両方**を検証（正常系だけでなく例外・エラーも）。
> - **バルクテスト**：1 件でなく **200 件**などまとめて処理してもガバナ制限内に収まるか確認。
> - **境界値・null・`System.runAs`** も意識する。
> - 例外を期待する場合は `try-catch` ＋ `System.assert` で検証。

> [!注意] バルクで動かないコードはテストで落ちる
>
> ループ内で SOQL/DML を書くと 200 件処理時にガバナ制限を超えて失敗する。**200 件規模のバルク検証**で設計ミスを本番前に発見できる。

## 練習問題とフラッシュカード（自己診断）

対話型の練習問題（**採点対象ではない**）と、**テストフレームワーク・実行・考慮事項**を扱うフラッシュカードがある。

> [!手順] 練習問題・フラッシュカードの進め方
>
> 1. 練習問題：シナリオを読み解答をクリック（複数正解あり）→ **[Submit]** で正誤と理由を確認。説明が長ければ **[Expand]**。
> 2. フラッシュカード：問題・用語を読み、カードをクリックで正解表示。矢印で前後へ移動。
