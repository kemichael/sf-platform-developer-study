## 主なトピック

この単元で確認するトピックは Apex クラス / Apex トリガー / 実行の保存順序。

> [!用語] Apex クラスとトリガー
>
> - **Apex クラス（Apex Class）**：関連する変数とメソッドをまとめた処理のかたまり（設計図）。再利用するロジックはクラスにまとめ、トリガーから呼び出すのが定石。
> - **Apex トリガー（Apex Trigger）**：レコードの保存イベント（登録・更新・削除など）に反応して自動実行されるコード。`before`（保存前）と `after`（保存後）のタイミングがある。

> [!用語] トリガーコンテキスト変数（Trigger Context Variables）
>
> トリガー内で使える特別な変数。**`Trigger.new`**（新しい値）、**`Trigger.old`**（更新前の値）、`Trigger.isBefore` / `Trigger.isAfter` / `Trigger.isInsert` などで、いつ・何が起きたかを判定する。

```apex
// Apex トリガーの基本形：取引先の登録・更新前に処理する
trigger AccountTrigger on Account (before insert, before update) {
    // Trigger.new：保存されようとしている新しいレコードの一覧
    for (Account acc : Trigger.new) {
        if (acc.Name != null) {
            acc.Name = acc.Name.trim();   // 前後の空白を除去（before なので保存に反映）
        }
    }
}
```

> [!ポイント] トリガーのベストプラクティス（頻出）
>
> - **1 オブジェクトにつきトリガーは1つ**（One Trigger per Object）。複数あると実行順が制御できない。
> - **ロジックはトリガー本体に書かずハンドラークラスに委譲**（Trigger Handler パターン）。
> - **必ずバルク化**（`Trigger.new` は常に複数件想定。ループ内 SOQL/DML 禁止）。
> - **`before` トリガーで同一レコードの値を変更**すれば追加 DML なしで保存に反映（`after` では明示的な update が必要）。

```apex
// ベストプラクティス：トリガーは入口だけ、処理はハンドラークラスへ委譲
trigger AccountTrigger on Account (before insert, before update) {
    AccountTriggerHandler.handleBeforeSave(Trigger.new);
}

public class AccountTriggerHandler {
    // ロジックはクラス側に集約（テストしやすく再利用しやすい）
    public static void handleBeforeSave(List<Account> accounts) {
        for (Account acc : accounts) {
            if (acc.Name != null) {
                acc.Name = acc.Name.trim();
            }
        }
    }
}
```

> [!例] before と after の使い分け
>
> - **before**：保存前に**同じレコードの値を変更・検証**したいとき（項目の整形、入力チェック）。
> - **after**：保存後に確定した ID を使って**関連レコードを操作**したいとき（子レコード作成、別オブジェクト更新）。新規レコードの `Id` は after でないと取得できない。

## 実行の保存順序（Order of Execution）

レコード保存時、Salesforce は**決まった順番**で各処理を実行する。これを理解していないと「フローで更新した値がトリガーで上書きされた」等の原因が分からなくなる。

> [!用語] 実行の保存順序（Order of Execution）
>
> 1 件のレコード保存時に、システム検証・before トリガー・入力規則・after トリガー・積み上げ集計・ワークフロー・フローなどが**どの順で走るか**を定めた流れ。試験最重要テーマ。

```text
   レコード保存時の実行順（おおまかな流れ）
   ┌─────────────────────────────────────────────┐
   │ 1. レコードをロード（既存値の読み込み）          │
   │ 2. システム検証（必須・データ型・外部キー など）  │
   │ 3. before トリガー実行                          │
   │ 4. システム検証を再実行＋入力規則の評価          │
   │ 5. レコードを保存（DBへ。まだコミットしない）     │
   │ 6. after トリガー実行                           │
   │ 7. 積み上げ集計・ワークフロー項目更新            │
   │ 8. レコードトリガーフロー（保存後フロー）         │
   │ 9. エスカレーション・ロールアップ等の後続処理     │
   │10. すべて成功したらコミット（ここで確定）         │
   └─────────────────────────────────────────────┘
```

> [!ポイント] 実行順で押さえる定番ポイント
>
> - **before トリガー → 入力規則 → after トリガー** の順。
> - **ワークフロー（やフロー）の項目更新で before/after トリガーが再実行される**ことがある（再帰の原因）。
> - 積み上げ集計・ロールアップは保存処理の後半で評価。
> - すべて成功して初めて**コミット**。途中で失敗すれば**全体がロールバック**。

> [!用語] 再帰（Recursion）とカスケード（Cascade）
>
> - **再帰**：トリガーが自分自身を間接的に再び呼び出すこと（例：after update が同レコードを update し再起動）。**static 変数のフラグ**で1回だけ実行するのが定石。
> - **カスケード**：ある処理が連鎖して別の処理（別オブジェクトのトリガー等）を次々起動すること。主従の親削除で子が削除され子のトリガーが走るのが典型。

```apex
// 再帰防止：static フラグで二重実行を防ぐ
public class TriggerControl {
    public static Boolean hasRun = false;   // 1トランザクション内で共有されるフラグ
}

trigger AccountTrigger on Account (after update) {
    if (!TriggerControl.hasRun) {
        TriggerControl.hasRun = true;       // 一度実行したらフラグを立てる
        // ここに1回だけ実行したい処理を書く
    }
}
```

## 試験対策：押さえておきたいポイント

> [!ポイント] Apex クラス・トリガー・実行順チェックリスト
>
> | 論点 | 要点 |
> | --- | --- |
> | One Trigger per Object | 1 オブジェクトにトリガーは1つ |
> | Trigger Handler パターン | ロジックはハンドラークラスへ委譲 |
> | バルク化 | `Trigger.new` は常に複数件。ループ内 SOQL/DML 禁止 |
> | before vs after | before=同レコード変更／after=ID 利用・関連操作 |
> | 実行順 | before トリガー → 入力規則 → after トリガー → 集計・フロー → コミット |
> | 再帰対策 | static フラグで1回だけ実行 |
> | カスケード | 親削除→子削除→子トリガー起動の連鎖に注意 |

> [!注意] よくある誤り
>
> - **after トリガーで `Trigger.new` のレコードを直接変更しても保存されない**（read-only）。変更したいなら before、または明示的に update。
> - **入力規則は before トリガーの後に評価される**。
> - **ワークフロー/フローの項目更新がトリガーを再起動**することがあり、再帰の温床になる。

> [!まとめ] この単元のまとめ
>
> - Apex は **トリガー→ハンドラークラス**に委譲し、**バルク化**して書く。
> - **before**＝同一レコードの変更・検証、**after**＝確定 ID での関連操作。
> - **実行の保存順序**は試験頻出。before→入力規則→after→集計/フロー→コミット。
> - **再帰**は static フラグで防ぎ、**カスケード**の連鎖に注意する。
