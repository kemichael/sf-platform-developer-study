SOQL vs SOSL vs DML

Apexにおけるデータ操作の3つの基本手段。それぞれ用途・構文・ガバナー制限が異なる。

項目 SOQL SOSL DML
目的 特定オブジェクトのレコード取得 複数オブジェクトをまたぐ全文検索 レコードの作成・更新・削除
基本構文 SELECT Id FROM Account WHERE... FIND 'keyword' IN ALL FIELDS RETURNING Account insert / update / delete / upsert
対象オブジェクト数 1クエリで1オブジェクト（子クエリは可） 複数オブジェクトを同時検索可 操作ごとに1オブジェクト
ガバナー制限 100回 / トランザクション 20回 / トランザクション 150回 / トランザクション
戻り値 List<sObject> List<List<sObject>> なし（Database クラスは SaveResult[]）
空文字列検索 WHERE 句で空文字列 = 不可 空文字列の FIND は不可 —
使用場所 Apex, Visualforce, LWC（wire） Apex のみ Apex のみ（ループ外推奨）

使い分けポイント

  • SOQL：条件（Id, 項目値）が明確なときに使用
  • SOSL：「キーワード」で複数オブジェクトを横断検索したいとき
  • DML：レコードを作成・変更・削除するとき（必ずリスト単位でまとめる）
試験ポイント
  • SOQL はループ内に書いてはいけない（Bulkification 違反 → Governor Limits 超過）
  • SOSL はテスト内で Test.isRunningTest() と組み合わせて固定値を返す実装が一般的
  • DML は 150回制限に注意。トリガーから再帰的に発火しないよう静的フラグを使う
  • SOSL の戻り値は List<List<sObject>>（二重リスト）であることに注意
比較 02

DML ステートメント vs Database クラスメソッド

同じDML操作を行う2つの記法。エラーハンドリングと部分コミットの対応が異なる。

項目 DML ステートメント Database クラスメソッド
記法例 insert recordList; Database.insert(recordList, false);
エラー時の挙動 DmlException をスロー → 全件ロールバック SaveResult[] を返す → エラーを個別確認可
部分コミット 不可（全成功 or 全失敗） 可（allOrNone=false を指定）
try/catch の必要性 必要（DmlException を catch） 不要（SaveResult でエラー確認）
upsert 外部ID指定 upsert list ExternalId__c; Database.upsert(list, ExtId__c.sObjectField, false);
ガバナー制限 同じ（150回 / tx） 同じ（150回 / tx）
試験ポイント
  • Database.insert(list, false) の false が allOrNone=false → 部分コミット可能
  • SaveResult.isSuccess() で成功/失敗を判定し、getErrors() でエラー内容を取得
  • DML ステートメントは失敗時に即座に DmlException がスローされる
  • どちらもガバナー制限（150回）は同じ。使い分けは「部分コミットが必要かどうか」
比較 03

非同期処理 4方式（@future / Queueable / Batch / Scheduled）

Apexで非同期処理を実現する4つの方法。ユースケースと制限が異なる。

項目 @future Queueable Apex Batch Apex Scheduled Apex
実装方法 アノテーション System.Queueable Database.Batchable System.Schedulable
起動方法 メソッド呼び出し System.enqueueJob() Database.executeBatch() System.schedule()
引数型 プリミティブ型のみ オブジェクト型可 —（QueryLocator または Iterable） —（引数なし）
戻り値 void のみ JobId（ID型） JobId（ID型） JobId（ID型）
ジョブのチェーン 不可 可（execute内で再 enqueue） 不可（finish内から起動は可） 不可
コールアウト callout=true で可 可 可（execute内） 不可（直接）
大量データ処理 不向き 不向き 最適（チャンク分割） 不向き
主要な制限 50回 / txn 呼び出し 同時50件キュー 同時5件アクティブ 100件スケジュール登録
ヒープサイズ 12MB 12MB 12MB 12MB
必須メソッド —（アノテーションのみ） execute(QueueableContext) start / execute / finish execute(SchedulableContext)

使い分けポイント

  • @future：シンプルな非同期・コールアウト。引数が少なくて済む場合
  • Queueable：オブジェクト型引数が必要、チェーン実行が必要、JobIDで監視したい場合
  • Batch Apex：数万〜数百万件の大量データを処理する場合
  • Scheduled Apex：定期実行（夜間バッチ起動など）が必要な場合（内部でBatchを呼ぶことも多い）
試験ポイント
  • @future は 引数にオブジェクト型不可（プリミティブ or プリミティブのコレクションのみ）
  • Batch Apex の start() は Database.QueryLocator または Iterable<sObject> を返す
  • Scheduled Apex の必須メソッドは execute(SchedulableContext sc) の1つだけ
  • Queueable から Queueable を enqueue するチェーン：同期テスト内では Test.stopTest() まで実行されない
  • Batch の最大チャンクサイズ：executeBatch(job, 200) で最大2000（Bulk API使用時）
比較 04

共有モード（with sharing / without sharing / inherited sharing）

Apexクラスに宣言する共有モード。実行ユーザーの共有ルール・OWDを適用するか否かを決める。

項目 with sharing without sharing inherited sharing
共有ルール適用 適用 無視 呼び出し元に依存
OWD 考慮 考慮する 無視 呼び出し元に依存
FLS（フィールドセキュリティ） 自動適用されない 自動適用されない 自動適用されない
全レコードへのアクセス 不可（権限内のみ） 可 呼び出し元に依存
キーワード省略時 inherited sharing と同等（呼び出し元の共有モードを継承）
推奨用途 一般ユーザー向けコード、UIコントローラ 管理操作、データ移行、システム処理 ユーティリティクラス、共通ライブラリ
試験ポイント
  • with sharing はFLSを自動適用しない。フィールドアクセス制限はコードで明示的にチェックが必要（Schema.sObjectType.Account.fields.Name.isAccessible()）
  • キーワードを省略すると inherited sharing 相当だが、明示的に書くことが推奨
  • global クラスは with / without sharing のどちらかを必須で宣言する
  • with sharing クラスが without sharing クラスを呼び出した場合、呼び出された側は without sharing で動作する
比較 05

List vs Set vs Map（Apex コレクション型）

Apexの3つのコレクション型。それぞれ特性が異なるため適切に使い分ける。

項目 List Set Map
順序 保持（インデックスアクセス可） 保持しない 保持しない（挿入順は一部保持）
重複 許可 不可（自動排除） キーは一意、値は重複可
アクセス方法 list[i] or list.get(i) set.contains(val) map.get(key)
初期化の例 new List<Account>() new Set<Id>() new Map<Id, Account>()
SOQLの結果格納 最適（直接代入可） 不可（直接代入不可） new Map<Id,Account>([SELECT...]) で一発
null 要素 可 1つだけ可 値はnull可、キーのnullは非推奨
典型的な用途 DML操作リスト、SOQL結果格納 重複排除、存在チェック（O(1)） Id→レコードの高速ルックアップ
試験ポイント
  • new Map<Id, Account>([SELECT Id, Name FROM Account]) はSOQL結果を直接Mapで受け取るショートカット
  • Set の contains() は O(1) でListの contains() はO(n)。存在チェックにはSetかMapが有利
  • List は add(), remove(index), size(), Set/Map は add()/put(), remove(), size()
  • Map の values() は List を返す、keySet() は Set を返す
比較 06

before トリガー vs after トリガー

Apex トリガーの2つの実行タイミング。何ができて何ができないかが重要。

項目 before トリガー after トリガー
実行タイミング DML 操作がDBに反映される前 DML 操作がDBに反映された後
Trigger.new の変更 可（変更が自動的にDBに反映） 不可（ReadOnly）
insert 時のId null（まだ採番されていない） 確定済み（採番後）
Trigger.old update/delete で利用可（insertはnull） update/delete で利用可（insertはnull）
DML 操作 可（再帰トリガーに注意） 可（関連レコードの操作に多用）
典型的な用途 値の自動設定・バリデーション・正規化 子レコード作成、関連オブジェクトの更新
試験ポイント
  • before insert では Trigger.new[i].Id は null（ID未確定）
  • before trigger で Trigger.new に値をセットするとき、update DML は不要（自動でDBに書き込まれる）
  • after trigger で Trigger.new を変更しようとすると System.FinalException が発生
  • 1オブジェクトに複数のトリガーを置くと実行順が不定になる → 1オブジェクト1トリガー推奨
比較 07

Custom Settings vs Custom Metadata Type（CMDT）

アプリケーション設定値を保存する2つの仕組み。デプロイ可能かどうかが最大の違い。

項目 Custom Settings Custom Metadata Type（CMDT）
デプロイ可否 不可（データはデプロイできない） 可（設定値ごとデプロイ可能）
パッケージング 不可 可
種類 Hierarchy 型 / List 型 1種類（レコードで管理）
SOQLカウント Hierarchy型はインスタンスメソッドで取得するとカウントされない SOQL 100回制限にカウントされる
Apex でのアクセス MySettings__c.getInstance() [SELECT ... FROM MyMDT__mdt]
テスト内での見え方 テストデータを作成しないと見えない テストコードからも自動的に見える（作成不要）
UI での編集 設定 → カスタム設定から編集可 設定 → カスタムメタデータ型から編集可
試験ポイント
  • デプロイや ISV パッケージに含めたい設定値は Custom Metadata Type を使う
  • Custom Settings の Hierarchy 型は「Profile 設定 → ユーザー設定」の階層で上書きできる
  • テスト内で Custom Settings の値が必要なときは insert new MySettings__c(...) を実行する必要あり
  • CMDT のレコードは Apex テスト内でも見えるため、テスト用セットアップが不要（テストデータ汚染リスクが低い）

Profile vs Permission Set vs Permission Set Group

Salesforceの権限設定の3階層。それぞれの役割と組み合わせ方を理解する。

項目 Profile Permission Set Permission Set Group
1ユーザーへの割り当て数 必ず 1つだけ 複数可（上限なし） 複数可
役割 基本権限のベースライン プロファイルへの追加権限 複数の Permission Set をまとめて管理
ログイン時間・IP制限 設定可 設定不可 設定不可
設定できる権限の幅 広い（タブ・アプリ・オブジェクト・FLSなど） 広い（プロファイルより狭い場合も） 複数 PS の束（権限そのものは持たない）
Muting Permission Set — — グループ内の一部権限を打ち消せる
推奨用途 ユーザー種別（営業・管理者等）の基本権限 特定機能を使うユーザーへの追加権限 複数 PS をロールに応じてセット管理
試験ポイント
  • Profile は必ず1つ割り当て必須。Permission Set は0個でも可
  • Permission Set は「プロファイルで持っていない権限を付与する」ことはできるが、「プロファイルの権限を剥奪」はできない
  • Permission Set Group（Spring '21〜）：複数のPermission Setを1つのグループとして一括割り当て可能
  • Muting Permission Set：PSG内の一部権限を打ち消す仕組み（PSGのみで利用可）

自動化ツール 3世代（Workflow Rule / Process Builder / Flow）

Salesforceの宣言的自動化ツール3世代の比較。現在はFlowへの集約が進んでいる。

項目 Workflow Rule Process Builder Flow
提供時期・状態 廃止予定（移行推奨） 廃止予定（移行推奨） 現役・推奨
トリガータイプ レコード変更のみ レコード変更・Platform Event 多様（レコード/スケジュール/画面/Platform Event/Apex）
ループ処理 不可 不可 可
Apex 連携 不可 可（@InvocableMethod） 可（@InvocableMethod）
サブフロー呼び出し 不可 不可 可（Flow内からFlowを呼び出せる）
利用可能なアクション フィールド更新・メール・タスク・アウトバウンドメッセージの4種のみ より多くのアクション（レコード作成・クイックアクション等） ほぼ全操作（作成・更新・削除・メール・外部・Apex等）
移行先 Flow Flow —
試験ポイント
  • Workflow Rule が使えるアクションは4種類のみ：フィールド更新・メール送信・タスク作成・アウトバウンドメッセージ
  • Process Builder・Workflow Rule は廃止予定（既存のものは動き続けるが新規作成は非推奨）
  • FlowのトリガーはRecord-Triggered / Schedule-Triggered / Screen / Platform Event / Apex-Triggered の5種
  • @InvocableMethod（@InvocableMethod(label='...')）を使うとApexメソッドをFlowから呼び出せる

デプロイ方法（Change Set / Salesforce CLI / Metadata API）

Salesforceのメタデータを組織間でデプロイする3つの方法。規模・自動化要件によって使い分ける。

項目 Change Set Salesforce CLI（sf） Metadata API
操作方法 Salesforce UI（ブラウザ） コマンドライン（sf / sfdx） SOAP API（プログラム）
バージョン管理連携 不可 可（Git との連携が容易） 可
CI/CD 自動化 不可 可 可
差分デプロイ 不可（手動選択） Source Tracking で自動検出 可（カスタム実装が必要）
パッケージ開発 不可 可（Unlocked Package 対応） 可
接続要件 接続組織（Deployment Connection）が必要 認証のみ（sf org login） 認証のみ
難易度 低（UIのみ） 中（CLI 操作） 高（API プログラミング）
推奨用途 小規模・非技術者向けの簡易デプロイ 開発プロジェクト標準（現在の推奨） ツール開発・高度な自動化

使い分けポイント

  • Change Set：小さな設定変更を素早くデプロイしたい非エンジニアや管理者向け
  • Salesforce CLI：Git を使ったチーム開発・CI/CD パイプライン構築（現在の標準）
  • Metadata API：独自のデプロイツールやアプリ内からプログラム的にデプロイしたい場合
試験ポイント
  • Change Set は一方向のデプロイ（Deployment Connection で接続先を事前設定が必要）
  • Change Set でデプロイする前に「検証（Validate）」を実行してテストを通過させる
  • Salesforce CLI の sf project deploy start でメタデータをデプロイ
  • Unlocked Package（アンロックパッケージ）はCLI/Metadata API でのみ作成可能。Change Set では不可
