# 教材ユニット スリム化仕様書（全エージェント共通）

すでに拡充済みの教材ユニット（Markdown）を、**視覚的にスキャンしやすくする**ため、要点中心に圧縮する。指定ファイルを**その場で上書き**する。

## 目的と分量
- 冗長さを削り、**要点が一目で追える密度**にする。
- 目標分量：**現状の約65〜75%**（読み込んだ行数を 0.7 倍程度に）。削りすぎ注意（用語・要点・図表は残す）。

## 維持するもの（削らない）
- 1行目の `# タイトル`、見出し階層（`##`/`###`）、`## 学習の目的` とその箇条書き。
- **コールアウト6種**（`[!用語]`/`[!例]`/`[!ポイント]`/`[!注意]`/`[!手順]`/`[!まとめ]`）。**記法厳守：マーカー行の次に必ず空の `>` 行**を入れる。
- Markdown の**表**、`` ```text `` の**図解**、`` ```apex `` 等の**コード例**。
- ハンズオン Challenge の**設定値・API名・項目名**、リソースのリンク、試験頻出ポイント。

## 圧縮の方針（ここを削る）
1. **冗長な地の文を圧縮**：同じ内容の言い換え・繰り返し・冗長な前置きを削り、一文を短く。箇条書きにできる説明は箇条書きへ。
2. **重複・類似コールアウトを統合**：同じ用語の重複解説、似た趣旨の `[!ポイント]` が複数あれば1つにまとめる。1ユニットに同種コールアウトが乱立している場合は**厳選**（重要なものを残す）。目安として用語解説は本当に初学者がつまずく語に絞る。
3. **用語解説は簡潔に**：各 `[!用語]` は1〜3文程度に。
4. **具体例は要点のみ**：`[!例]` は核心が伝わる最小限に。
5. **コードのコメント**を冗長なら簡潔化（ただしコード本体の動作は変えない）。
6. 自明な説明・冗長な「まとめの繰り返し」を削除。

## 厳守事項
- 全文**日本語**。コードの**動作・Challenge設定値・API名は不変**。
- コールアウト記法（**マーカー行の次に空の `>` 行**）を必ず守る。先頭は `#` 見出し。
- 情報の**取捨選択は可**だが、試験に必要な要点・用語・図表・手順は残す。意味が変わる削除はしない。
- 結果は**同じファイルパスに上書き保存**。書き込み後に読み直し、見出し・コールアウト記法・フェンスが壊れていないか自己確認する。Write が確定しなければ再実行。

## 返却
担当した各ファイルについて「パス / 圧縮後の行数（元→新）/ 統合・削除した主な内容」を簡潔に報告する。
