# 数字まとめ（数値・上限値）

Salesforce Platform デベロッパー試験で **「具体的な数値」が正解の決め手になる項目** を 1 ページに集約しました。ガバナ制限・各種上限・保持期間・配点など、暗記が必要な数値をカテゴリ別に整理しています。試験直前の最終確認に活用してください。

> [!ポイント] このページの使い方
>
> - 左サイドバーのカテゴリから、目的の数値群へジャンプできます。
> - **同期／非同期で値が変わる**ガバナ制限は特に頻出です。両方を必ず覚えてください。
> - 教材本文に明記された数値を主表記とし、教材に記載のない一般知識は最後の「補足」にまとめています。
> - 各数値の詳しい背景は[学習ノート本編](index.html)を参照してください。

> [!注意] 数値は丸暗記より「なぜその値か」を意識
>
> 数値だけを暗記すると、似た値（50,000 と 2,000、200 と 2,000 など）で混同しがちです。「同期は厳しめ・非同期は緩め」「件数系か行数系か」といった**意味のグループ**で捉えると、ひっかけ選択肢に強くなります。

## 1. Apex ガバナ制限（同期／非同期）

同期コンテキストと非同期コンテキスト（@future・Queueable・Batch・Schedulable）では上限が変わります。**非同期側のカウントは同期側とは独立**しています。試験で最頻出の領域です。

### 同期・非同期で値が変わる主要制限

| 項目 | 同期 | 非同期 | 補足 |
| --- | --- | --- | --- |
| SOQL クエリ発行回数 | **100 回** | **200 回** | 1 トランザクションあたり。非同期は倍 |
| SOQL で取得できる総レコード数 | **50,000 行** | 50,000 行 | 同期・非同期とも同じ |
| DML ステートメント発行回数 | **150 回** | 150 回 | 同期・非同期とも同じ |
| DML で処理できるレコード数 | **10,000 行** | 10,000 行 | ※後述の表記ゆれ注意 |
| ヒープサイズ | **6 MB** | **12 MB** | 非同期は倍 |
| CPU 時間 | **10,000 ms（10 秒）** | **60,000 ms（60 秒）** | 非同期は 6 倍 |

> [!ポイント] 「100 / 200」と「6 / 12」と「10秒 / 60秒」はセットで暗記
>
> 倍率が項目ごとに違う（SOQL は ×2、ヒープは ×2、CPU は ×6）のがひっかけポイント。「非同期 SOQL = 200」「非同期ヒープ = 12MB」「非同期 CPU = 60秒」と数値そのもので覚えるのが安全です。

### 同期・非同期で共通の制限

| 項目 | 数値 | 補足 |
| --- | --- | --- |
| SOSL クエリ発行回数 | **20 回** | 1 トランザクションあたり |
| SOSL で返る最大レコード数 | **2,000 行** | 全オブジェクト合計。SOQL の 50,000 と混同注意 |

> [!注意] DML 処理レコード数の表記ゆれ
>
> 教材内に表記ゆれがあります。`14_Apexトリガー` / `15_非同期Apex` では「**10,000 行**」、`11_データベースと.NET` では「合計 50,000 件まで」と記述されています。後者は SOQL の取得行数（50,000）と混同した可能性のある記述です。**正規のガバナ制限としては「DML = 10,000 行」を主表記**としてください。

## 2. 非同期 Apex（Batch / Queueable / future / Schedulable）

| 項目 | 数値 | 補足 |
| --- | --- | --- |
| トリガーが 1 回で受け取るレコード数 | **最大 200 件** | 200 件単位のバッチで動作。ガバナ制限はバッチ単位 |
| Batch Apex デフォルトスコープサイズ | **200 件** | `execute()` に渡る件数。`Database.executeBatch` の第 2 引数で変更可 |
| Batch（QueryLocator）取得最大レコード数 | **5,000 万件** | Iterable 使用時は通常の SOQL 制限 |
| Batch で処理できる最大レコード数 | **1 億 5,000 万件** | データローダーの上限と同じ値 |
| Queueable の enqueue 上限 | **1 トランザクションに最大 50** | |
| Queueable のチェーニング | **1 親 → 1 子** | future は連鎖不可。Queueable は子を 1 つだけ追加可 |
| Queueable チェーン深度（Developer Edition） | **5** | Developer / Trial 組織の制限 |
| Schedulable で一度にスケジュールできるジョブ数 | **100 件** | スケジュール実行内では同期コールアウト不可 |
| Apex Flex キュー 保留可能ジョブ数 | **最大 100 件** | Batch ジョブ専用。FIFO、保留中は順序入替可 |
| Flex キュー 同時実行ジョブ数 | **最大 5 件** | 組織ごと |

> [!ポイント] 「200」が出てくる場所を整理
>
> トリガーのバッチ（200）／Batch Apex のデフォルトスコープ（200）／SOQL for ループのバッチ（200）はすべて **200**。一方でプラットフォームイベントのトリガーは **2,000** 件単位です（後述）。「通常は 200、プラットフォームイベントだけ 2,000」と覚えると混同しません。

## 3. テスト関連

| 項目 | 数値 | 補足 |
| --- | --- | --- |
| 本番リリースに必要なコードカバー率 | **75% 以上** | 全テスト合格 ＆ 全トリガーが最低 1 行カバーされることも必須。最頻出 |
| テスト実行時のメール送信 | **0 件（送信されない）** | テストコンテキストでは実際の送信は行われない |
| `@IsTest` クラスのコードサイズ | **6 MB に含まれない** | テストクラスは組織のコードサイズ上限の対象外 |

> [!ポイント] Test.startTest() / stopTest() は「ガバナ制限のリセット」
>
> `Test.startTest()` ～ `Test.stopTest()` の区間内では**同期ガバナ制限が新しくカウント**され、区間内でエンキューした非同期処理は `stopTest()` で同期的に実行されます。「75%」と並ぶテスト分野の頻出ポイントです。

## 4. SOQL / SOSL / DML

| 項目 | 数値 | 補足 |
| --- | --- | --- |
| SOQL for ループのバッチサイズ | **200 件** | DB アクセスごとに 200 件ずつ返す |
| SOQL ドット表記の参照階層（子 → 親） | **最大 5 レベル** | 子から親方向にたどれる段数 |
| SOSL 検索文字列の最大長 | **20,000 文字** | |
| OFFSET の上限 | **2,000 件** | 大量データのページングには不向き |
| merge で統合できる最大レコード数 | **3 件** | 1 件を残し 2 件を統合・削除 |
| merge 対応の標準オブジェクト | **4 種類** | 取引先・取引先責任者・リード・ケース |
| 削除レコードのごみ箱保管期間 | **15 日** | この間は `undelete` で復元可 |

## 5. データ管理（インポート／エクスポート）

| 項目 | 数値 | 補足 |
| --- | --- | --- |
| データインポートウィザード 最大件数 | **50,000 件（5 万件）** | ブラウザベースのウィザード |
| データローダー 最大件数 | **1 億 5,000 万件** | クライアントアプリ。Batch Apex の上限と同じ |
| エクスポート ウィークリー周期 | **7 日ごと** | Enterprise / Performance / Unlimited 対象 |
| エクスポート マンスリー周期 | **29 日ごと** | Professional / Developer 対象 |
| エクスポートファイルの保持期間 | **48 時間** | 経過後に自動削除 |
| チェックボックスのインポート値 | **1（あり）／ 0（なし）** | |

> [!ポイント] インポート 50,000 と ローダー 1.5 億をセットで
>
> 「ウィザード = 50,000 件まで、それ以上はデータローダー（1 億 5,000 万件）」という使い分けがそのまま問われます。エクスポート周期は「ウィークリー 7 日／マンスリー 29 日／ファイル保持 48 時間」の 3 点セットで暗記。

## 6. 数式・入力規則・データモデリング

| 項目 | 数値 | 補足 |
| --- | --- | --- |
| クロスオブジェクト（スパン）数式の参照階層 | **最大 10 階層** | 親方向にドットでたどれる段数 |
| 積み上げ集計の集計種別 | **4 種類** | COUNT / SUM / MIN / MAX |
| 1 つの子オブジェクトが持てる主従関係 | **最大 2 つ** | ジャンクションオブジェクトで多対多を表現 |
| レコード ID の長さ | **15 桁（大小区別あり）／ 18 桁（区別なし）** | API では 18 桁が安全 |

> [!豆知識] レコード ID の先頭 3 文字（プレフィックス）
>
> ID の先頭 3 文字はオブジェクトを示します。**001 = 取引先、003 = 取引先責任者、006 = 商談、500 = ケース**。デバッグ時にどのオブジェクトの ID か即判別できます。

## 9. 試験そのものの配点

試験の各セクションの配点（出題比率）です。配点の大きい「プロセスの自動化とロジック」を重点的に学習しましょう。

| セクション | 配点 |
| --- | --- |
| プロセスの自動化とロジック | **28%（最大）** |
| 開発者の基本事項 | **27%** |
| ユーザーインターフェース | **25%** |
| テスト・デバッグ・リリース | **20%** |

> [!豆知識] 教材に記載のない試験情報
>
> 合格スコア・問題数・試験時間・受験料は教材本文には明記されていません（一般には合格ライン約 68%・60 問・105 分・200 USD が目安とされますが、公式の最新情報を確認してください）。
