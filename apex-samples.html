01

Apex 基礎

プロセス自動化・ロジック ★★★★★

Apexのデータ型、コレクション、制御フローは試験の基礎。すべての問題に関わるため確実に押さえること。

データ型と変数Apex
コレクション（List / Set / Map）Apex
制御フロー（if / for / switch on）Apex

試験のポイント

  • Integer は整数、Decimal は小数（通貨）、Long は大きな整数
  • List は順序あり・重複あり、Set は順序なし・重複なし、Map はキー→値
  • Apex の switch on は Java と異なり fall-through がない
  • Null 参照に対してメソッド呼び出しすると NullPointerException が発生する
02

SOQL / SOSL

データモデリング・管理 ★★★★★

データ取得の要。リレーションクエリ、集計関数、バインド変数（インジェクション対策）、動的SOQLまで幅広く出題される。

基本クエリ・フィルタ・並び替えApex
リレーションクエリ（親→子 / 子→親）Apex
集計クエリ（GROUP BY / HAVING）Apex
動的 SOQL / SOSLApex

試験のポイント

  • バインド変数（:variable）を使うとSOQLインジェクションを防げる
  • 子→親リレーションは ドット記法（Contact.Account.Name）
  • 親→子はサブクエリ（SELECT Id, (SELECT Id FROM Contacts) FROM Account）
  • WITH SECURITY_ENFORCED でFLS・CRUDをSOQLレベルで強制できる
  • SOSLは複数オブジェクトを横断検索、SOQLは単一オブジェクト
03

DML 操作

プロセス自動化・ロジック ★★★★☆

insert / update / upsert / delete / undelete の使い方と、エラーハンドリングを許容する Database.* メソッドの使い分けが頻出。

標準 DML ステートメントApex
Database.* メソッド（部分成功・SaveResult）Apex

試験のポイント

  • 標準DML（insert list）は allOrNone=true（1件エラーで全件ロールバック）
  • Database.insert(list, false) はエラーレコードをスキップして残りを保存
  • upsert は外部IDフィールドを指定しない場合はSalesforce IDで判定
  • DML制限は1トランザクションで 150回、対象レコードは 10,000件
  • Database.Savepoint でロールバックポイントを設定できる
04

トリガー開発

プロセス自動化・ロジック ★★★★★

試験最重要トピック。バルク対応（ループ内SOQL/DML禁止）、ハンドラーパターン、Trigger.*コンテキスト変数の使い方を完全習得すること。

AccountTrigger — トリガー本体（ハンドラーへ委譲）Apex
AccountTriggerHandler — バルク対応ハンドラーApex

試験のポイント

  • Trigger.new（挿入・更新後の値）/ Trigger.old（更新・削除前の値）
  • Trigger.newMap / Trigger.oldMap は Id → sObject のマップ
  • before trigger で Trigger.new のフィールドを直接変更できる（DML不要）
  • after trigger では Trigger.new は読み取り専用
  • ループ内でSOQL/DMLを実行すると101件超でガバナ制限エラー → 必ずループ外で一括処理
05

ガバナ制限

テスト・デバッグ ★★★★☆

Salesforce のマルチテナント環境を守る制限。「何件まで？」という数値と、制限を回避するパターンが問われる。

NG — ループ内でSOQL実行
OK — ループ外でSOQL一括実行
Limits クラス — 使用量をリアルタイムで確認Apex

主要ガバナ制限（同期トランザクション）

  • SOQL クエリ: 100回 / SOQL 取得レコード: 50,000件
  • DML ステートメント: 150回 / DML 対象レコード: 10,000件
  • CPU 時間: 10,000ms / ヒープサイズ: 6MB
  • Callout: 100回 / Callout タイムアウト: 120秒

例外処理

プロセス自動化・ロジック ★★★☆☆

try/catch/finally の構造、例外の種類（DmlException / QueryException など）、カスタム例外クラスの実装が出題される。

try / catch / finally と例外の種類Apex
カスタム例外クラスApex

試験のポイント

  • カスタム例外は extends Exception で定義する
  • finally ブロックは例外の有無に関わらず必ず実行される
  • DmlException.getDmlMessage(0) で最初のエラーメッセージを取得
  • LWC / Aura からのエラーは AuraHandledException でラップしてスローする
07

非同期 Apex

プロセス自動化・ロジック ★★★★★

@future / Batch / Queueable / Scheduled の4種類の使い分けが頻出。それぞれのインターフェース・制約・実行方法を覚えること。

@future — 非同期・コールアウト対応Apex
Batch Apex — 大量レコードの一括処理Apex
Queueable Apex — ジョブチェーン対応Apex
Scheduled Apex — cron でスケジュール実行Apex

試験のポイント

  • @future: 引数はプリミティブ型か Set<Id>。sObjectは渡せない
  • Batch: start / execute / finish の3メソッド。デフォルトチャンク200件
  • Database.Stateful を implements するとバッチ間で変数を保持できる
  • Queueable はジョブのチェーン（連結）が可能。テスト時は1ジョブのみ
  • Scheduled: cron 式で最大同時100ジョブ。スケジュール済みジョブは削除できない
08

テスト

テスト・デバッグ・デプロイ ★★★★★

本番へのデプロイには75%以上のコードカバレッジが必要。@testSetup、Test.startTest()、HttpCalloutMock、アサーションの使い方が必須。

テストクラスの基本構造（@testSetup・正常系・異常系・バルク）Apex
HttpCalloutMock — 外部APIのモックApex

試験のポイント

  • テストクラスは @isTest アノテーションを付与し、本番コードをカウントしない
  • Test.startTest() / Test.stopTest() でガバナ制限をリセットし、非同期処理を完了させる
  • @testSetup で共有テストデータを1回だけ作成（各テストは自動ロールバック）
  • コールアウトのテストは必ず Test.setMock() で HttpCalloutMock を設定
  • テストデータは Test.isRunningTest() などで本番データと分離する
09

オブジェクト指向（OOP）

プロセス自動化・ロジック ★★★☆☆

アクセス修飾子、abstract / virtual クラス、インターフェース、継承（extends）の構文と動作原理が問われる。

インターフェース・抽象クラス・継承・オーバーライドApex

試験のポイント

  • abstract クラスはインスタンス化不可。abstract メソッドはサブクラスで必ず実装
  • virtual クラスはインスタンス化可能で、virtual メソッドはオーバーライド可能
  • インターフェースはメソッドシグネチャのみ定義（実装なし）
  • Apex はシングル継承（extends 1クラスのみ）、インターフェースは複数実装可能
  • アクセス修飾子: private < protected < public < global

セキュリティ

Salesforce 基礎 ★★★★☆

with sharing / without sharing の使い分け、CRUD・FLSの検証、Security.stripInaccessible の活用。セキュリティ問題は毎回出題される。

with sharing / without sharing / inherited sharingApex
CRUD / FLS チェック・Security.stripInaccessibleApex

試験のポイント

  • with sharing: OWD・共有ルールを適用（推奨デフォルト）
  • without sharing: 共有設定を無視（管理者専用処理など最小限に限定）
  • inherited sharing: 呼び出し元クラスの設定を引き継ぐ
  • 共有キーワードを省略すると without sharing と同等（危険）
  • Security.stripInaccessible はアクセス不可フィールドを自動で除去する（推奨）
